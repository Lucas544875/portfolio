# 非ユークリッドの窓(ポータル) — プロトタイプ#5

`portfolio-concept.md` の「ガラスの向こう側」「階層を下る = 別の空間へ潜る」という筋から
派生した、Portal(Valve)風の非ユークリッドな継ぎ目トリックの単体プロトタイプ。
[`../prototype/`](../prototype/)(KanjiVGなぞり)・[`../prototype-rain-window/`](../prototype-rain-window/)(雨の窓)・
[`../prototype-shards/`](../prototype-shards/)(ガラスの破片)とは独立した、完全に別のWebGL2アプリ。

フレームワーク無し、ビルド不要。

## 実行方法

```sh
cd prototype-portal
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## コンセプト

1 枚の平らなガラス板(ポータル矩形)がある。正面から見ると、その板の奥は手前の部屋
(Room A)が「そのまま空間的に続いている」ように見える——ポータル下端では床のラインが
同じ高さのまま奥へ延び、一見ふつうの窓越しの景色に見える。ところが実際にそこに映って
いるのは、座標変換でつながれた**別の部屋**(Room B)で、向きも回っていればオブジェクトも
別物・巨大。物理的に連続した空間ではない。この「入口の見た目の位置」と「代数的な出口の
位置」のミスマッチが非ユークリッド感(= ありえない奥行き)を生む。サイト全体の
「ガラスの向こうに、ありえない深さがある」という感触の実験。

## できていること

- **単一パスのレイマーチだけで継ぎ目トリックを実現**(屈折用の render-to-texture 不要)。
  1 本のレイマーチループの途中で座標系を乗り換える:
  1. まず入口ポータル矩形との交差を**解析的**に求める(平面 ∩ 矩形の交差 `t`)。これは
     「レイ位置のポータル平面への符号付き距離が `+` から `-` へ反転する点を検出する」ことと
     数学的に等価だが、交差が一意に閉じた式で決まるため、グレージング角でレイが平面を
     何度も横切って**詰まる**危険が無い。
  2. Room A を、その交差 `t` まで通常どおりレイマーチする。途中で Room A のジオメトリ
     (床・箱・ガラス額縁)に当たればそこで確定(= 額縁の外を通るレイは Room B に漏れず、
     Room A の壁などをそのまま描く = **クリッピング**)。
  3. 何にも当たらずにポータル矩形の内側の平面へ到達したら、そのレイの**位置と方向を
     剛体変換(回転 + 平行移動)**で Room B のワールド座標系へ写し、残りのレイを Room B の
     (形も色も違う)SDF でマーチし続ける。
- **剛体変換の中身**(`main.js` 冒頭に手計算の検証コメントあり、起動時に `console.assert`
  でも確認):
  - フレーム行列 `M = [u | v | n]`(正規直交・右手系)を入口 A・出口 B それぞれに作り、
    回転 `R = M_B M_A^T`、`p' = CENTER_B + R (p - CENTER_A)`、`d' = R d`。
  - `R` は入口 A の局所座標を出口 B の局所座標へ 1:1 で写すので、「A 矩形上で局所オフセット
    `(a,b)` の点は、B 矩形上でも同じ `(a,b)` の点へ(向きだけ変えて)写る」。
  - 出口 B は入口 A から**遠く離れた任意の場所**(`CENTER_B = (40,0,40)`)に、**垂直軸まわりに
    40° 回した**姿勢で置いている。垂直軸まわりの回転だけにしているのは、床の高さ `y` が変換で
    保存され、ポータル下端の床が「同じ高さで」Room B へ連続して見えるから(まず連続に錯覚
    させ、よく見ると向きが回っている & オブジェクトが別物…と気づかせる順を狙う)。
- **Room A と Room B の描き分け**(継ぎ目が「別の場所への窓」と読めるように):
  - Room A = サイトのトーンの**青灰色・冷たい光**(結露ガラスのパレット)。床・数個の箱・
    ガラス額縁。
  - Room B = **暖色(テラコッタ)・逆側からの暖かい主光源**の別空間。オブジェクトは
    `OBJ_SCALE = 3.0` 倍の**巨大な角柱・浮遊する大きな球・奥の壁**で、ありえない奥行きを強調。
- **ガラス板らしさ**:ポータルの縁は薄いガラス額縁(box の差で作る矩形リング。フレネルの
  縁光 + うっすら青いティント)。開いているポータルの奥(Room B)にも、わずかなティントと
  視線が寝るほど強まるフレネル縁光を重ね、「ガラス越しに見ている」感を出す。
- **横・裏から見たときの挙動(意図的な設計判断)**:これは物理的な穴ではなく 1 枚の板なので、
  真横で「薄い縁」だけになると嘘がバレる。そこでポータルが**開く**(= Room B が見える)のは
  「カメラが板の表側に居て、かつ視線が板に対して十分立っている(非グレージング、
  `|dot(rd, n)| > GRAZE_MIN`)」ときだけ。**表からでもグレージング角、または裏側から見た
  ときは、内側を不透明な曇りガラス板(Room A の環境を反射)として描く**。これにより横・裏
  からは「ただのガラス板」に見え、正面からだけ奥の別空間が開ける。
- **無限ループ防止**:座標系の乗り換えは 1 レイあたり**最大 1 回**(`CROSSING CAP = 1`)。一度
  Room B に入ったら二度と乗り換えないので、どんな角度でも有限ステップで終わる。Room A /
  Room B の各マーチも固定ステップ数(`MAX_STEPS`)・最大距離(`MAX_DIST`)で打ち切る。
- カメラはロード時にポータル正面のやや斜めから遠 → 近へ**自動ドリーイン**し、継ぎ目が
  読みやすい構図に寄る。ユーザーがドラッグした瞬間に自由なオービットへ引き継ぐ
  (プロトタイプ #1〜#4 と同じ「自動再生 → ハンドオフ」)。
- モバイル(`matchMedia("(pointer: coarse)")`)では内部解像度・ステップ数を自動的に減らす。

## 設計上のメモ・チューニング

- **符号反転の毎ステップ検出ではなく、解析的な平面 ∩ 矩形交差にした。** タスクの指定は
  「毎ステップ符号を見る or 平面交差テスト」のどちらでもよいという条件で、後者を選んだ。
  平面は 1 枚・レイは直線なので交差は高々 1 点に閉じた式で決まり、(1) グレージングでの
  詰まりが原理的に起きない、(2) 交差点を厳密に求められる(補間誤差なし)、という利点がある。
  この解析交差は「符号付き距離が `+→-` に反転する点」と等価。
- **床の高さを A/B で揃える(`FLOOR_Y` 共通)ことが錯覚の肝。** 変換が `y` を保存する垂直軸
  回転なので、ポータル下端の床が途切れず連続して見える。ここがズレると一目で「別レイヤー」
  とバレて連続感が出ない。
- **ポータルは「くぐる扉」ではなく「覗く窓」なので、180° の反転は入れていない。** 歩いて
  くぐる Portal(ゲーム)なら出口の表から反対向きに出るため上方向まわりに 180° 回すが、
  ここは覗き込む窓なので、A の局所座標をそのまま B の局所座標へ写す(左は左のまま)direct な
  対応にしている。
- 額縁は `outer box − inner opening` の差で矩形リングにしている。中央付近を通る内側のレイは
  額縁 SDF に当たらず平面へ到達して乗り換え、縁へ向かうレイは額縁に先に当たる、という
  振り分けが SDF だけで自然に起きる。

## 視覚的に未検証な点(ブラウザでの目視が必要)

この環境にはヘッドレスブラウザ / スクリーンショット手段が無く、**実際の描画は未確認**。
以下は数式・ロジックとしては通しているが、人間が実機で見て調整すべき可能性が高い点:

1. **`GRAZE_MIN`(0.16)での「開く / 閉じる」の切り替えがハードカット。** オービットで
   グレージング角を跨いだ瞬間に、Room B の眺めが曇りガラスへパッと切り替わる。グリッチ的に
   見えるようなら、遷移帯で `marchRoomB` の結果と `shadeClosedPane` をスムーズにブレンド
   する必要がある(現状は分岐で即切り替え)。値そのものも要調整。
2. **クリッピングがグレージング角で額縁の外へ Room B を漏らす / 逆に内側に暗い縁を作る
   可能性。** 内側の矩形ぎりぎりを通るレイは額縁 SDF に近づいてスフィアトレースが遅くなり、
   `MAX_STEPS` 内に平面へ届かず背景にフォールバックして、開口の内側に細い暗いスジが出る
   ことがあり得る。`MAX_STEPS` / 額縁寸法(`FRAME_W`, `HALF`)の調整が要るかもしれない。
3. **Room B の姿勢(`TURN = 40°`)と配置が、実際に覗いたとき不格好な角度になる可能性。**
   40° の向き回転でオブジェクトが画面の端に寄りすぎる / 奥の壁が斜めに切れて見える等。
   角柱・球・壁の位置(`fwd*距離`, `rgt*横`)と `TURN` は実機で構図を見て詰める前提。
4. **Room B の「奥の壁」を軸並びの薄い box で近似している。** Room B は回転している空間なので、
   壁の法線が `fwd` と揃っておらず、覗く方向によっては板が斜めに浮いて見える恐れがある。
   気になるなら壁もローカル軸へ回すか、床 + 奥行きフォグで代替する。
5. **カメラの初期構図(`CAM_YAW_START`, `CAM_PITCH_START`, `CAM_DIST`)が最適か未確認。**
   ポータルが画面中央に程よく収まり、継ぎ目が読める画角になっているかは目視前提。
6. **色・明るさのバランス(トーンマップ後)。** Room A の青灰色と Room B の暖色のコントラスト、
   額縁フレネルの強さ、ビネットは、reinhard + ガンマ後に実際どう出るか未確認。
7. **`FOV`(1.4)の広角感。** 広すぎると周辺が歪み、狭すぎるとポータルが小さく見える。
