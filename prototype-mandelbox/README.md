# マンデルボックスのレイマーチ — プロトタイプ #12

参考: https://hausdorff-dimension.netlify.app/mandelbox.html

マンデルボックス(Tom Lowe による IFS 系フラクタル、Rrrola の距離推定式)を
レイマーチで描く。参考サイトは6自由度で自由に飛び回れる一方、常にほぼ同じ
縮尺で見て回るため「マンデルボックスが無限に細かい構造を持つ」という一番
大事な性質が体感しづらい。このプロトタイプは自由なフライトを捨て、代わりに
「同じ1点へ、対数スケールで一定速度でひたすら潜り続けるカメラ」に絞ることで、
その性質を正面から見せることに賭けている。フレームワーク無し、ビルド不要。

## 実行方法

```sh
cd prototype-mandelbox
python3 -m http.server 8000
# → http://localhost:8000 をブラウザで開く
```

自動で「全体像を見せる → 同じ1点へズームイン → フェードして次の1点へ」を
延々と繰り返す。ドラッグで視点を微調整、ホイールでズーム速度を加速・減速
できる(移動そのものの向きは変えられない——後述「カメラ」参照)。

## できていること

- Rrrola型のマンデルボックス距離推定(box-fold → ball-fold →
  scale+translate を固定回数反復)を、ズーム段数に関わらず**定数コスト**で
  評価する実装(後述「計算コストがズーム段数に依存しない理由」)。
- 「全体像(overview)→ 同じ1点への対数ズーム(dive)→ フェードして
  次の点を選び直す(fade)」の3段サイクル。参考サイトと違い、常に
  「今どれだけ拡大しているか」を見失わない(後述「3段のサイクル」)。
- ズーム倍率の対数値(`10^x.xx×`)とプログレスバーを常時表示する HUD。
  「見た目が複雑に見え続けている」ことを感覚だけでなく数値でも裏付ける。
- 配色・ライティングは参考サイト(`mandelbox.frag`)のマテリアル・
  アンビエント/ディフューズ・白熱光(incandescence)・グロー・ガンマ補正を
  移植し、見た目のトーンを合わせている(後述「着色・ライティング」)。
- 表面判定のイプシロンをレイの進行距離とピクセルの投影サイズに比例させる
  「pixel-projected epsilon」により、遠景でも細部が潰れて見えない
  (後述「遠景でのっぺりする問題」)。
- WebGL2のGLSLには無いハードウェア倍精度を、float32のペア(hi/lo)で
  再現する df64(double-float)演算で補い、ズームの下限を大幅に押し下げた
  (後述「df64(double-float)による倍精度」)。
- 潜る先の1点は GPU 側と全く同じ式を JS 側に倍精度で移植し、毎サイクル
  スフィアトレースして選び直す(後述「潜る先の選び方」)。
- Node.js で DE・精度限界・df64演算・潜る先の探索成功率・状態機械の遷移を
  検証し、さらに headless Chromium(Playwright, `--use-gl=swiftshader`)で
  実際にページを描画してスクリーンショットで見た目を確認した
  (後述「検証方法」)。

## マンデルボックスの距離推定(DE)

反復のたびに次の3手順を行う([Rrrola/Buddhi 系の実装](https://www.fractalforums.com/)
でよく使われる形):

1. **box-fold**: 各成分を `clamp(x, -1, 1)*2 - x` で置き換える。`[-1,1]` の外に
   出た分だけ折り返す。
2. **ball-fold**: `r² = dot(z,z)` として、`r² < MINR2` なら
   `z *= FIXEDR2/MINR2`、`MINR2 ≤ r² < FIXEDR2` なら `z *= FIXEDR2/r²`
   (球面反転)。
3. **scale+translate**: `z = z*SCALE + p`(`p` は最初の入力点=レイの現在位置。
   Mandelbrot の `z→z²+c` の `c` と同じ役割)。

`dr`(導関数の大きさ)を同じ反復で追跡し(`dr = dr*|SCALE| + 1`、fold で
掛けた係数も同様に `dr` に掛ける)、最終的に `length(z) / abs(dr)` を距離推定
として返す。パラメータは `SCALE=2.5, MINR2=0.09, FIXEDR2=1.0` (`prototype-mandelbox/main.js`
の `CONFIG` 参照)。バウンディング半径は Node.js で12方向に対してレイマーチし
実測(概ね4.6〜5.0、`ITER=14` 固定)。

## 計算コストがズーム段数に依存しない理由

Mandelbrot(2D)の深いズームでは、境界に近づくほど「何回反復すれば発散する
か」が増えていくため、深く潜るほど `max_iter` を増やす必要がある。
マンデルボックスの DE はこれと違い、**ITER を固定したまま**ズーム段数を
何桁上げても成立する。理由は距離推定そのものの定義にある——
`length(z)/|dr|` は「今の反復回数の中で、周囲のどれだけ小さい近傍が
まだ同じ側に潰れずに残っているか」を `dr` の増大に押し込めて返す量なので、
ズームで見えている微細構造は「反復を増やしてやっと見える」ものではなく、
**同じ ITER 回の反復から返る値の中に、対象点をどれだけピンポイントに
指定できるか(=浮動小数点の精度)次第で既に折り畳まれて入っている**。
つまり「参考サイトが常に一定の縮尺なのは絵作りの都合ではなく、実は
計算コストの方が本質的にズーム段数に依存しない」という、このフラクタル
特有の性質がある。このプロトタイプは、その性質を逆手に取って「精度が
許す限界まで、コストを増やさずにひたすら潜り続ける」設計にした。

## 遠景でのっぺりする問題(pixel-projected epsilon)

レイマーチで「表面にヒットした」と判定するイプシロンを、以前は
`uMaxDist(=現在のカメラ距離*定数) * 係数` という、画面全体で一定の
値にしていた。overview のようにカメラが遠い(=`uMaxDist`が大きい)
場面では、このイプシロンが絶対値として粗くなりすぎ、自己相似な
フラクタルが本来どのスケールでも持っているはずの細かい皺や穴が
埋もれて「のっぺり」した見た目になっていた。

修正後は、イプシロンをレイの進行距離 `t` とその時点で1ピクセルが
投影される角度サイズ(`pixelAngle = 1/(FOCAL*解像度)`)の積に比例させる
(`surfEps = max(SURF_EPS_MIN, t * pixelAngle * SURF_EPS_PIXEL_MULT)`)。
これは「そのピクセルが画面上で解像できる最小サイズ」を表面判定の基準に
する、レイマーチングでは定番の手法で、遠くのオブジェクトほど1ピクセルが
広い範囲をカバーする分イプシロンも自然に粗くなり、近くのオブジェクトは
逆に厳しく判定されるため、無駄なステップを使わずに画面全体で
「見える範囲の限界」まで細部を残せる。

## 精度の壁(実測)——float32単体だとどこで崩れるか

固定コストで潜れると言っても、無限に潜れるわけではない。float32 は
目標点 `p0`(O(1) の座標)から見て相対 `1e-6〜1e-7` 程度離れたオフセット
までしか安定に解決できない(23bit 仮数部の ULP)。これを机上の空論に
せず、`Math.fround` で float32 演算を丸めてシミュレートした DE と倍精度
DE を比較して実測した(検証用に一時的に書いたスクリプトはスクラッチ
パッドに置いただけで、このリポジトリには含めていない):

| オフセット半径 | 倍精度 vs float32 相対誤差 |
|---|---|
| 2.5e-4 | 0.06% |
| 3.9e-5 | 8.6e-3% |
| 2.4e-4 | 0.12% |
| 6.1e-5 | 0.46% |
| 3.8e-6 | 3.6%(ここから float32 のノイズフロアが支配的になる) |
| 3.0e-8 | 0.05%(残差がほぼ一定値に収束=真の距離ではなく `p0` 自身の残差) |

float32単体のままだと、この表の通りズームの下限は `1e-5` オーダーが
限界になる。

## df64(double-float)による倍精度

WebGL2 の GLSL には(desktop OpenGL の `GL_ARB_gpu_shader_fp64` のような)
ハードウェアの倍精度型が無い。そこで、1つの数値を float32 のペア
`(hi, lo)` で表現し、Dekker/Knuth の2Sum・2Prod アルゴリズムで四則演算を
実装する「double-float(df64)」を GPU 側に自前で実装した(`main.js` の
`twoSum`/`twoProd`/`dfAdd`/`dfMul`/`dfDiv` など、および `vec3` 版の
`DF3`/`df3*` 関数群)。

**なぜ座標だけ df64 にすれば足りるか**: 精度が壊れる直接の原因は
「目標点 `p0`(O(1))の近傍にある、`dist` スケールの微小オフセットを
どこまで区別できるか」という一点に尽きる。マンデルボックスの DE 反復
そのもの(`SCALE`・`MINR2`・`FIXEDR2`・`FOLD` との演算)は、この座標さえ
正確に持ち回れれば float32 のままで構わない——反復ごとの `z = z*SCALE + p`
という増幅(`SCALE=2.5` を14回)が、O(1)の中に埋もれた微小オフセットを
毎回2.5倍ずつ引き上げてくれるので、**入力さえ正確なら出力は自然と
見える桁まで増幅される**。そのため `mapDE` は座標 `z`・`p` だけを
`DF3`(df64のvec3版)で持ち、`dr` や `SCALE` などのスカラー係数は
float32のままにしてある。

**実測した精度の伸び**: Node.js 上で GLSL と同じ df64 演算(`Math.fround`
で float32 丸めをシミュレート)を再現し、真の倍精度(JS の `number`)を
リファレンスにして「df64 で表現した点がどのオフセット半径まで
`offset=0` の場合と区別できるか」を測った(`p0` にわずかに違う方向へ
オフセットを与え、DE の出力が `offset=0` の値と一致してしまう
=情報が失われた、とみなす)。5点の異なる座標・方向で実測したところ、
df64 が飽和し始めるオフセットは概ね `1e-12`〜`1e-14` だった。float32
単体(`1e-6`〜`1e-7`)に対して、**約6桁の改善**になる。

**「offset+p0」を素のfloat32で先に合成してはいけない**: 最初の実装では
`p = p0(df64) + (camOffset + rd*t)`(`camOffset+rd*t` を素のfloat32加算で
先に1つのベクトルへ合成してから `p0` へ補正加算)としており、これだと
`camOffset` と `rd*t` それぞれが「`dist` スケールの中でのfloat32精度
(24bit)」しか持てず、df64で `p0` に足し込む段階ではもう手遅れだった
(実機・Node.js双方で `10^5〜10^9×` 付近から構造が消えることで発覚)。
修正後は `ro = p0 + camOffset` を先に df64 の補正加算で1回だけ確定させ、
各レイマーチステップでは `p = ro + rd*t`(`rd*t` は単独では打ち消し合いが
起きない量なので素のfloat32のままdf64へ補正加算してよい)としている。

**潜る先自身の精度も足並みを揃える必要がある**: df64 でどれだけ座標を
正確に持ち回っても、`pickTarget()` が探す `p0` 自身が isosurface から
ズレていては意味がない。旧実装の `jsRaymarch` は `d < 1e-5 * max(1,t)`
というかなり緩いしきい値でヒット判定していた(残差 `jsDE(p0)` は実測で
`~4e-6`)。これは旧 `DIST_MIN=2.5e-5` の頃はちょうど同じオーダーで
無視できたが、`DIST_MIN` を大きく下げた今では `p0` 自身の残差の方が
支配的になり、潜るほど全く見当違いの構造に着地したり完全に外れたりする
原因になっていた。しきい値を `PICK_TARGET_HIT_EPS = 1e-13` まで締めて
解決した(JS側は倍精度なので、しきい値を締めても収束に必要な反復回数は
数十ステップ程度しか増えない——実測)。

**自動首振り(`AUTO_YAW_SPEED`)も精度と無関係に効いてくる**: pixel-projected
epsilon(後述)導入前は表面判定が粗く、カメラの向きが `p0` の方向から
多少ズレていても周辺の粗い構造に「引っかかって」何かしら描画されていた。
精密なイプシロンにした後は、dive の後半でカメラの自動首振りが累積して
大きくズレると、レイが本当に何にも当たらなくなり画面が真っ黒になる
(Node.js上の同一ロジック再現で、ヨー角が`90°`に近づくにつれてヒット数が
`53/81`→`0/81`まで単調に落ちることを確認)。dive 全体を通した首振りが
大きくなりすぎないよう `AUTO_YAW_SPEED` を引き下げて対処した
(`main.js` の `CONFIG.AUTO_YAW_SPEED` 参照)。

**実用上のズーム下限**: df64単体の理論上のノイズフロア(`1e-12`〜`1e-14`)
まで攻めると、画面中心から外れたレイ(=画面の大部分)では、旧
`DIST_MIN(2.5e-5)`を下回るあたりから徐々にヒットがまばらになっていく
(スクリーン全域を1200x800相当・8x8点グリッドで実測。中心の1本だけなら
`1e-10`でも誤差1%程度で安定にヒットするが、画面全体では話が別)。
「開始から終了まで画面全体で高精細を保つ」ことを理論上の最大深度より
優先し、`DIST_MIN = 5e-5` という、旧実装とほぼ同じ深さに留める控えめな
値を採用した。ズームの**深さ自体は伸ばしていない**が、df64・
`pickTarget()`の精度・pixel-projected epsilon による改善で、同じ深さでも
旧実装のように痩せて消えることなく最後まで安定して描画できるように
なっている。より深く潜りたい場合は「次のステップ」参照。

**二重精度でも摂動法は要らない理由**: Mandelbrot の深いズームで使われる
摂動法は「基準軌道を高精度で計算し、そこからのずれだけを低精度で追う」
手法だが、これは `z²+c` のような非線形反復が基準点ごとに大きく発散
経路を変えるために必要になる。このプロトタイプではカメラの移動を
「`p0` を通る1本の直線上を前後するだけ」に限定しているため、精度の
危険因子は常に「ワールド座標そのもの」1つだけに絞れる。df64はこの
座標表現の精度を底上げするだけで、DE の反復構造自体は変えていない。
参考サイトのような自由な6自由度移動を諦めた分だけ、精度の問題も
単純になっている。

## 3段のサイクル

1. **overview(既定4秒)**: `dist` を `OVERVIEW_DIST` に固定し、フラクタル
   全体をバウンディング球の外から見せる。
2. **dive(既定23秒)**: `dist` を `OVERVIEW_DIST` から `DIST_MIN` まで
   **指数的に**(=対数目盛りで等速に)縮める。HUD のズーム倍率表示が
   一定速度で桁を刻んでいくのはこのため。
3. **fade(既定1.1秒)**: 画面を暗転させ、暗転が最大になった瞬間に次の
   潜る先を選び直して `dist` を `OVERVIEW_DIST` に戻す。ユーザーからは
   「暗転の裏でシーンが差し替わる」だけに見え、ズーム開始点が唐突に
   ワープするような違和感がない。

カメラは終始「`p0` を通る1本の直線上を前後するだけ」で、オービットは
しない(`buildCameraFrame()` 参照)。ドラッグによる視点操作は、この基準
方向に対する小さな yaw/pitch の上乗せとして最後に加える——移動の軌道
そのものには影響しない。これにより精度の危険因子が `dist` の1変数だけに
なり、扱いが単純になる(前節参照)。

## 潜る先の選び方

`main.js` の DE をそのまま JS 側に倍精度(`number`型)で移植し
(`jsDE`/`jsRaymarch`)、バウンディング球の外側のランダムな点から
「ほぼ原点方向+わずかなジッター」へレイを飛ばして命中点を採用する
(`pickTarget()`)。GPU 側でズームするのと全く同じ式を CPU 側でもう一度
倍精度でなぞるだけなので、毎サイクル1回・数百ステップ程度の追加コストは
無視できる。Node.js で200回試行したところ、フォールバック(既知の
ハードコード点)に頼った回数は0回、ほぼ全て初回の乱数方向で命中した
(検証用の使い捨てスクリプトで確認。リポジトリには含めていない)。
命中点ごとに局所形状が変わるため、サイクルごとに違う場所へ潜れる。

## 着色・ライティング

当初は反復中に到達した最小の `|z|²`(orbit trap)を深藍〜暖色の混合率に
使う独自配色だったが、参考サイト(`hausdorff-dimension.netlify.app`の
`mandelbox.frag`)に見た目を近づけるため、以下をそのまま移植した:

- ベースカラーは単色の茶(`BASE_COLOR = (0.454, 0.301, 0.211)`、参考
  サイトのマテリアル `BROWN`)。
- アンビエント(`0.7 * BASE_COLOR`)+ディフューズ(`1.1 * diff * BASE_COLOR`、
  ライト方向 `normalize(2,1,1)`)。スペキュラ・シャドウ・大域照明・反射は
  参考サイトでも実効的に寄与が無いため省略。
- 白熱光(incandescence): 座標軸 `±GLOW_RADIUS` 上の6点を中心とした
  オレンジ色(`(1.0, 0.501, 0.2)`)の点光源。参考サイトの `incCenter`/
  `incRadius`(共に`2.0`)を、参考サイトのカメラ距離(`11`)に対する比率
  (`≒0.18`)でこちらの `OVERVIEW_DIST` にスケールした。
- グロー: レイマーチで消費したステップ数の割合(`iterate`)に応じて
  同じオレンジを加算。反復を多く使った=細部が密な箇所ほど白熱する
  参考サイトの演出で、「潜るほど自己相似の細部が見える」という
  このプロトタイプのコンセプトとも相性がいい。
- 背景は完全な黒(参考サイトの `SAIHATE` マテリアル)。
- ガンマ補正は `pow(col, 2.2)`(逆数ではない)という参考サイト独自の
  実装をそのまま踏襲。暗部を強く締めてコントラストを上げる。

## 踏んだ罠: フェーズラベルが更新されない

実装直後、headless Chromium でズーム倍率が `10^5×` を超えても HUD の
フェーズ表示が「全体像を確認中…」のまま変わらないという不具合に気づいた。
原因は `updateCycle()` の状態機械で、`phase` を `"overview"` と `"fade"`
の2値としてしか更新しておらず、`cycleT` が `ORBIT_DURATION` を超えて
dive 相当の時間に入っても `phase` を `"dive"` に書き換える分岐が
無かったこと(カメラの `dist` 自体は `cycleT` から直接計算しているため
実害は無く、HUD の文字列だけがずれていた)。`else` 節の中で
`phase = cycleT <= ORBIT_DURATION ? "overview" : "dive"` を明示的に
書き足して解決した。

## 検証方法

このプロトタイプは Playwright にキャッシュされていた Chromium
(`~/.npm/_npx/*/node_modules/playwright`、実行ファイルは
`~/.cache/ms-playwright/`)を `--use-gl=swiftshader` で起動し、実際に
`python3 -m http.server` 越しにページを開いてスクリーンショットで見た目を
確認しながら作った。加えて:

- DE・バウンディング半径・`pickTarget()` の探索成功率は Node.js の
  倍精度実装で単体検証した。
- float32 の精度限界は `Math.fround` で丸めをシミュレートし、倍精度版
  との相対誤差を複数スケールで比較して実測した。
- サイクルの状態機械(overview→dive→fade→overview、`pickTarget()` が
  ちょうど1サイクルに1回呼ばれるか)は、`main.js` と同じロジックを
  切り出した Node.js スクリプトで `dt=1/60` を積算するシミュレーションで
  検証した。

  (これらの検証スクリプトはいずれもリポジトリ外のスクラッチパッドに
  書いた使い捨てで、`prototype-mandelbox/` 配下には含めていない。)
- 上記の単体検証だけでは「実際のページでフェーズラベルが正しく遷移する
  か」は分からなかったため(前節の罠はこの経路で見つかった)、headless
  Chromium 上で `#zoomReadout`/`#zoomPhase` の DOM テキストを実際に
  ポーリングし、overview→dive→overview の遷移を目視ではなくテキストで
  確認した。

  なお headless + swiftshader (CPU ソフトウェアレンダリング)環境では
  1フレームの描画自体に時間がかかるため、`dt` のクランプ(`Math.min(0.05, …)`)
  の影響でサイクルの進行が実時間よりかなり遅く見える(実測で概ね
  2.6〜3倍程度)。これは実際の GPU 上ではほぼ発生しない headless 環境
  特有の現象で、`dt` クランプ自体は「タブが非アクティブな間に一気に
  時間が飛ぶ」ことを防ぐ安全策としてそのまま残している。

## 次のステップ

- `DIST_MIN` を df64 の理論上のノイズフロア(`1e-12`〜`1e-14`)まで
  さらに攻めたい場合は、画面全域(特に画面端の斜めのレイ)でも安定して
  収束するようステップ数やイプシロンの安全率をさらに詰める余地がある
  (README「df64(double-float)による倍精度」の「実用上のズーム下限」
  参照。現状は画面全体の高精細さを旧実装と同程度の深さの範囲で優先し、
  `5e-5`に据え置いている)。
- df64 は「float32のペア」止まりだが、さらに `triple-float`(3つの
  float32)や、`p0` 周辺だけを再スケールする「正規化(renormalization)」
  を組み合わせれば、もう数桁分ズームを伸ばせる可能性がある。
- ソフトシャドウ・AO を追加すると陰影の情報量が増えて微細構造がさらに
  見やすくなる可能性がある(今回はコストを抑えるため見送った)。
- df64はDE反復1回あたりの演算量を大きく増やすため(実測でheadless
  swiftshader上のフレームレートが目に見えて低下)、`ITER`を段階的に
  減らす、あるいは`dist`が十分大きい間だけfloat32単体の高速パスに
  切り替える、といった性能最適化の余地がある。
