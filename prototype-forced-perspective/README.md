# 強制遠近法 — プロトタイプ#5

エイムズの部屋(Ames room)系の「強制遠近法(forced perspective)」の錯視を扱う単体プロトタイプ。
[`../prototype/`](../prototype/)(KanjiVGなぞり)・[`../prototype-rain-window/`](../prototype-rain-window/)(雨の窓)・
[`../prototype-shards/`](../prototype-shards/)(ガラスの破片)とは独立した、完全に別のWebGL2アプリ。

フレームワーク無し、ビルド不要。

ある1点の "スイートスポット" カメラから見ると、本当は大きさも距離もまったく違う4つの球が
「同じ大きさで横一列に並んでいる」ように見える。カメラがその1点から離れると普通の遠近法が
効きはじめ、本当の大きさ(実半径 0.3 対 8.1 = 27倍差)と奥行き(距離 3 対 81 = 27倍差)が
露わになって錯視が崩れる。`portfolio-concept.md` の「沈んでいく階層 / 奥行きを体感させる」
モチーフとも通じる、奥行きをテーマにした一枚。

## 実行方法

```sh
cd prototype-forced-perspective
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## できていること

- **錯視の構成(このプロトタイプの核心。数学的に厳密)**
  - スイートスポットのカメラを原点に置き前方 `(0,0,-1)` を見る。物体 i の中心を
    「カメラからの真の距離 d_i」・水平角 α_i の方向、`center_i = d_i * (sin α_i, 0, -cos α_i)`
    に置く。単位方向 × 距離なので `|center_i| = d_i` が厳密に成り立つ
  - 半径 r_i の球がカメラに張る**見かけの角半径は `asin(r_i / d_i)`**。よって `r_i / d_i` を
    全物体で同じ定数 `k` にすれば、d_i がどれだけ違っても見かけの大きさが完全に一致する。
    これが「同じ大きさに見える」の定義そのもの
  - 本プロトタイプは `k = RATIO = 0.1` に固定:

    | 物体 | 実距離 d | 実半径 r | r/d | 水平角 α | 中心座標 | 見かけの角半径 |
    |---|---|---|---|---|---|---|
    | A(琥珀) | 3  | 0.3 | 0.1 | -24° | (-1.220, 0, -2.741) | asin(0.1)=5.739° |
    | B(シアン) | 9  | 0.9 | 0.1 | -8°  | (-1.253, 0, -8.912) | 5.739° |
    | C(若草) | 27 | 2.7 | 0.1 | +8°  | (3.758, 0, -26.737) | 5.739° |
    | D(藤) | 81 | 8.1 | 0.1 | +24° | (32.946, 0, -73.997) | 5.739° |

    見かけの角半径はどれも `5.739°`(直径 `≈11.48°`)で完全に等しい。一方で実寸も距離も27倍差
  - 水平角は 16° 等間隔なので、隣り合う球の見かけの隙間は `16 - 11.48 = 4.52°`。スクリーン上で
    重ならない=**手前の球が奥の球を隠さない**(奥行きが違っても遮蔽が起きず、4個が横一列に読める)
  - これらの r_i・center_i は `main.js` 冒頭の `OBJECT_TABLE` の d_i・α_i から
    `r = RATIO*d` / `center = d*方向` として**導出**している。比率が全物体で一致していることが
    コード上でも自明になり、手打ちの数値ミスが入らない
- **カメラが動いても物体の位置・スケールは一切変えない。** 種明かしは「スイートスポットでだけ
  見かけが一致する」という正直な透視投影の帰結としてのみ起こる(物体側は無加工)
- レイマーチ(sphere tracing)で4つの球SDFを描く。完全球なので法線 `normalize(hit - center)` が
  厳密に求まり、見かけの大きさの検証に余計な近似が入らない
- 背景は暗いグラデーション + まばらな星のみ。**床グリッド等の奥行き手がかりをあえて置かない**
  ことで、スイートスポットでの錯視を最大限保つ(手がかりは動かした時の視差だけになる)
- 各球は識別しやすいよう別々の色。「藤色の球が実は巨大で遠い」と崩れる瞬間に追跡できる
  ようにするため(材質のリアルさより可読性を優先)
- カメラは注視点 `target=(0,0,-9)` の周りをオービットし、`yaw=pitch=0, dist=9` のとき
  カメラ位置がちょうど原点(スイートスポット)に来る
- ページを開くと `AUTOPLAY_HOLD`(2.6秒)スイートスポットで静止して第一印象を作り、その後
  yaw を振り子のように振って「錯視の成立→崩壊→再成立」を繰り返す。ドラッグ(オービット)/
  ホイール(dolly)した瞬間にユーザー操作へ引き継ぐ(プロトタイプ#1〜#4と同じ設計)
- モバイル(`matchMedia("(pointer: coarse)")`)では内部解像度・レイマーチのステップ数を減らし、
  さらに `FOV` を下げて縦長画面でも4個すべてが画面に収まるようにする

## 設計の根拠・チューニングのメモ

- **d_i を「カメラからの真の(ユークリッド)距離」に定義したのが肝。** 中心を
  `d_i * 単位方向ベクトル` に置くことで `|center_i| = d_i` が厳密に成り立ち、見かけの角半径
  `asin(r_i / d_i)` が**近似なしで**全物体一致する。もし d_i を「前方(z)軸への射影距離」と
  定義すると、横にずれた物体ほど真の距離が伸びて見かけが微妙にずれてしまう。この違いは
  α が大きい物体D(24°)で効いてくる
- 見かけの水平位置 `ux = FOV * tan(α)` は FOV に比例するが、**見かけの角半径 `asin(k)` は FOV に
  依存しない**。つまり FOV(ズーム)を変えても錯視の比率は保たれ、画面上の拡大率だけが変わる。
  これを使ってモバイルでは FOV を下げ、画面幅に4個を収めている
- 16° 等間隔にしたのは、見かけ直径 11.48° に対して十分な隙間(4.52°)を確保しつつ、4個が
  横並びに読める範囲に収めるため。間隔を詰めすぎると隣の球と重なって遮蔽が起き、錯視の
  「横一列」感が壊れる
- `DIST_MIN = CAM_DIST = 9` にして、スイートスポットより手前に dolly できないようにしている
  (原点=錯視の一致点。そこを突き抜けると物体Aの内側に入ってしまう)

## 視覚的に未検証な点

この環境にはヘッドレスブラウザもスクリーンショット取得手段も無く、**WebGLの実出力は一度も
目視できていない**。数値・数式は詰めたが、以下は人間がブラウザで見て初めて確定できる:

- **見かけの説得力(最重要)**: r/d 比が全物体で 0.1 に一致するのは数学的に厳密だが、
  「実際に人間の目に同じ大きさ・横一列に並んで見えて錯視として成立するか」は未検証。
  角半径 5.739°(FOVに対する画面占有)が小さすぎ/大きすぎる可能性があり、その場合は `RATIO`
  と `FOV`、および α の間隔を一緒に見直す必要がある
- **カメラ運動のパラメータは当て推量**: `AUTOPLAY_HOLD`(静止時間)・`AUTOPLAY_YAW_AMP`
  (振れ幅)・`AUTOPLAY_YAW_SPEED`(速さ)は「これくらいで崩れて見えるだろう」の勘。
  振れ幅が小さいと崩壊が地味、大きすぎると第一印象の前に崩れて見える恐れがある。
  `WHEEL_SENSITIVITY` / `DRAG_*_SENSITIVITY` の操作感も未確認
- **モバイル FOV(0.95)の妥当性**: 縦長・横長さまざまなアスペクト比で4個すべてが画面内に
  収まるかは実機未確認。極端な縦長では外側の物体A/Dが画面端で切れる可能性がある
- **配色と背景**: 4色の識別しやすさ、星や背景グラデーションの明るさ、ビネットの強さは
  すべて未目視。特に物体が背景に埋もれず、かつ光りすぎない balance は要調整
- **レイマーチの品質**: `MAX_STEPS` / `SURF_EPS` が遠い物体D(距離~90)の輪郭を滑らかに
  描けているか、球のシルエットにステップ由来のノイズが出ていないかは未確認
