# 溶ける文字 — プロトタイプ #4

`portfolio-concept.md` のブレストから派生した、「流体の中に文字が現れて溶けていく」
単体プロトタイプ。[`../prototype/`](../prototype/)(KanjiVGなぞり)・
[`../prototype-rain-window/`](../prototype-rain-window/)(雨の窓)・
[`../prototype-crystal/`](../prototype-crystal/)(レイマーチ結晶)とは独立した、
完全に別のWebGL2アプリ。fluidエンジンの土台部分は`../prototype/`の実装を流用している。

フレームワーク無し、ビルド不要。

## 実行方法

外部アセット読み込みは無いが、`<script type="module">` は `file://` では
CORSでブロックされるため、他のプロトタイプと同様にローカルサーバーが必要。

```sh
cd prototype-fluid-text
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## できていること

- `../prototype/` と同じ stable-fluids(移流・渦度閉じ込め・圧力投影)エンジンの上に、
  「なぞる」代わりに「文字全体を一気にスタンプする」新しい注入方式を実装
- **文字データはKanjiVGではなく、明朝体フォントをCanvas 2Dで直接描画したもの。**
  `"Hiragino Mincho ProN", "Yu Mincho", "Noto Serif JP", "MS Mincho", serif` の
  フォントスタックで`fillText`し、そのアルファチャンネルをアルファマスクの
  テクスチャとして使う。外部ファイル・fetchは一切無く、文字の追加も文字列を
  足すだけで済む(詳細は下記「KanjiVGから明朝体フォントへの切り替え」)
- 1サイクルを「出現(0.4秒・一気にスタンプ)→保持(2.2秒・水流をほぼ止めて形を守る)
  →溶解(3.6秒・水流を強めて渦に押し流す+dissipationも強める)→間(0.7秒・無地の水)」
  の4フェーズで構成し、これを4文字(水・光・夢・深)をシャッフルした順で無限に繰り返す
- ページを開けば何もしなくても自動でループする。ポインタ/タッチで水面に触れると、
  渦とごく薄いインクの滲みを追加できる(任意の演出、無くても成立する)
- モバイル(タッチ主体の端末)を`matchMedia("(pointer: coarse)")`で判定し、
  fluidシミュレーションの解像度・圧力反復回数を下げるフォールバックを実装

## KanjiVGから明朝体フォントへの切り替え

最初はKanjiVGのストロークパスSVGをラスタライズして文字の型に使っていたが、
「文字データとしてKanjiVGではなく明朝体フォントを使いたい」というフィードバックを
受けて`createGlyphTexture()`に置き換えた。

- Canvas 2Dの`fillText`でオフスクリーンcanvasに白文字を描き、そのアルファ
  チャンネルをそのままインクの型として使う。KanjiVGの時は「SVGをfetch→
  名前空間パッチ→Blob化→`<img>`でラスタライズ」という非同期の変換が必要
  だったが、フォント描画は同期的に一発で済むため、`Promise.all`での読み込み
  待ちも丸ごと不要になった(起動直後にテクスチャを作り終えられる)。
- 副産物として、KanjiVGが持っていた筆順の数字ラベル(`kvg:StrokeNumbers_*`)を
  取り除く処理も不要になった — 明朝体フォントの字形だけが最初から手に入るため。
- `assets/kanjivg/`のSVG・fetch関連コードは全て削除し、依存ファイルが無い
  完全に自己完結した実装になった。
- 明朝体は仮想ボディの中心と字面の視覚的重心がわずかにずれるため、
  `fillText`の描画位置をキャンバス中央から少し下にオフセットして
  光学的な中央寄せをしている。

## 「保持してから溶ける」アークの作り方

指数関数的な減衰(dissipation)だけでは、"しばらく形を保ってから溶ける" という
知覚的な弧は作れない(dissipationは対数的に見れば常に一定速度で減っていくため、
「保持」と「溶解」を体感で分けられない)。そこで意図的に2段階へ分けた:

- **保持フェーズ:** `DENSITY_DISSIPATION_HOLD`(0.9997、ほぼ減衰しない)に加えて、
  アンビエントな水流の強さも`AMBIENT_FORCE_HOLD_MULT`(0.1)でほぼ止める。
  最初は水流を止めずに実装したところ、渦度閉じ込め(curl)がすぐに文字を細い筋へ
  歪ませてしまい、「保持」に見えなかった — dissipationだけでなく、水流そのものを
  弱めないと形は保てない、という発見。
- **溶解フェーズ:** `DENSITY_DISSIPATION_DISSOLVE`(0.965、一気に減衰)に切り替え、
  同時にアンビエント水流も`AMBIENT_FORCE_DISSOLVE_MULT`(1.8)で強める。
  「量が減る」のと「渦に押し流されて広がる」のが同時に起きることで、
  ただフェードアウトするより「溶け込んでいく」感が出る。
- 出現直後の「勢いで水を押しのける」演出用のvelocity splatも当初は入れていたが、
  保持フェーズの立ち上がりで不要な歪みを生んでいたため削除した。

## チューニングのメモ

- ポインタでのドラッグ操作は、最初「ごく薄いインクの滲み」のつもりで実装したところ
  実際には強すぎ、少し引っかくだけで画面が真っ白に飽和した。連続するsplatが重なる上、
  渦度閉じ込めがインクを細い筋に濃縮するため、見た目のガウシアン半径から想像するより
  ずっと低い色の強さ(最終的に`USER_DYE_INTENSITY: 0.022`)にしないと飽和する。
  `../prototype/`・`../prototype-rain-window/` の「半径は直感よりずっと小さい値が
  正しい」という教訓と同じ形の罠が、今回は「色の強さ」でも起きた。
- dissipation系のuniformは元々「60fpsの1フレームあたり」の値として定義されている
  ため、フレームレート差で体感速度が変わらないよう`Math.pow(base, dt * 60)`で
  実時間補正している(`../prototype-rain-window/`で得た教訓の再利用)。
  出現→保持→溶解→間の秒数指定が実際の秒数として機能するために必須。
- `../prototype/` のCURL(渦度閉じ込めの強さ)は9だったが、この用途ではそのままだと
  保持フェーズでも渦が強すぎて文字がすぐ崩れた。6まで下げている。

## 分かったこと・検証方法について

- Playwright(Chromiumヘッドレス)でデスクトップ・モバイル双方のビューポートを
  スクリーンショットし、出現→保持→溶解→次の文字、という一連のサイクルを実際に
  時間を進めながら確認した。ヘッドレスのデフォルト設定では
  `EXT_color_buffer_float`を使うfloatレンダーターゲット主体のこのアプリ(および
  `../prototype/`)がまれに`CONTEXT_LOST_WEBGL`を起こしたが、
  `--use-gl=angle --use-angle=swiftshader`等のフラグを足すことで安定して再現・
  確認できた。これはヘッドレス特有のソフトウェアレンダラーの問題で、
  実装側の不具合ではないと判断している(`../prototype/` の既存コードでも同様の
  事象が再現したため)。
- 実機のモバイル端末での実際のfps・タッチ操作の感触は未確認。
