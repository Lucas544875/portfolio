# ガラスの破片 — プロトタイプ#4

`portfolio-concept.md` のブレストから派生した、たくさんのガラス破片が浮かぶ空間の単体プロトタイプ。
[`../prototype/`](../prototype/)(KanjiVGなぞり)・[`../prototype-rain-window/`](../prototype-rain-window/)(雨の窓)・
[`../prototype-crystal/`](../prototype-crystal/)(結晶レイマーチ)とは独立した、完全に別のWebGL2アプリ。

フレームワーク無し、ビルド不要。

## 実行方法

```sh
cd prototype-shards
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## できていること

- 正八面体1個をジオメトリインスタンシングで100〜150個複製し、「空間中にたくさん浮かぶガラスの破片」を表現
  - プロトタイプ#3(結晶)はレイマーチで1つの結晶を描いたが、同じ手法で個数を増やすと
    (内部を進んで屈折の出口を探す)コストが破片数に比例して跳ね上がるため、多数の破片には
    ラスタライズ+インスタンス描画の方が適している
- 各破片は同じ正八面体の頂点データを共有しつつ、頂点シェーダー内でインスタンスごとに
  - 各頂点を放射方向にランダムジッター(不揃いな破片シルエット)
  - 軸ごとの非等方スケール(細長い破片らしさ)
  - ランダムな自転軸・位相・速度(個々に違うタイミングでゆっくり自転)
  を適用しており、追加のジオメトリ生成コードなしで見た目の異なる破片群になっている
- 法線は頂点属性を持たず、`dFdx`/`dFdy` によるスクリーンスペース微分でその場から求める。
  インデックス共有ジオメトリのままで三角形単位の完全にフラットな(=鋭いファセットの)陰影になる
- 反射・屈折の配分と色収差は、クリスタルガラス(鉛ガラス)の物理量から一貫して導出している
  - 破片ごとに緑(d線)の屈折率 `n_d` を実際の鉛クリスタルの実測レンジ(`1.54〜1.56`。
    ソーダ石灰ガラスのn≈1.5より高い)でランダムに振る
  - 反射率はこの `n_d` からSchlick近似(`F0 = ((n-1)/(n+1))²`、`F0 + (1-F0)(1-cosθ)⁵`)で
    導出し、fresnel-mixで`reflect()`方向の環境(プロシージャルな深宇宙+3つのスタジオ
    光源風の光点)と屈折色をブレンドする
  - R/G/Bそれぞれの屈折率は、アッベ数(分散の指標。値が低いほど分散が強い)の定義式
    `V_d = (n_d - 1) / (n_F - n_C)` を鉛クリスタルの代表値(`≈38`。ソーダ石灰ガラスは
    `≈59`で、鉛クリスタルの方が分散が強く"炎"のようなきらめきが出やすい)で逆算し、
    それぞれ別方向で実際に`refract()`する。**この3方向のズレそのものが色収差**
    (ファセットの縁に出る虹色のにじみ)になる
- **屈折の先には「本当にその場にある破片」が映る。** 毎フレーム2パス描画にし、
  1パス目で通常の絵をオフスクリーンのテクスチャにcaptureしておき、2パス目(本番)では
  屈折方向にワールド座標を少し進めた点を再度カメラ射影してスクリーンUVを求め、
  そのcaptureテクスチャをそのまま覗く(screen-space refraction)。プロシージャルな
  背景だけを屈折させていた最初のバージョンは「奥の破片が透けない」ため不透明感が薄く
  チープに見えていたが、この修正で屈折の先に隣の破片がちゃんと映り込むようになった
  (1バウンス分の近似。captureテクスチャ自体はプロシージャル背景のみを屈折させているため、
  屈折の屈折…という無限再帰は起きない)
- 全ての破片を不透明(alpha=1)・深度テスト/書き込み有効で描画している。多数の半透明
  ジオメトリを正しくブレンドするには本来ソートが要るが、反射・屈折の色自体が
  「背景や他の破片がねじれて透けて見える」効果を担っているため、ソート不要のまま
  十分ガラスらしく見える
- ページを開くとカメラが自動で空間全体をゆっくり周回し、ユーザーがドラッグした瞬間に
  自由なオービット操作へ引き継ぐ(プロトタイプ#1〜#3と同じ「自動再生→ハンドオフ」の設計)
- モバイル(`matchMedia("(pointer: coarse)")`)では内部解像度・破片数を自動的に減らす

## 分かったこと・チューニングのメモ

- **最初、破片が大きすぎて「浮かぶ破片の群れ」ではなく「岩の塊」に見えた。**
  サイズ・伸縮の乱数レンジ(`CONFIG.SIZE_MAX` / `STRETCH_MAX`)を空間の広がり
  (`FIELD_MAX_RADIUS`)に対して十分小さく取らないと、1〜2個の巨大な破片が画面を
  支配してしまう。「たくさんの小さな破片」を狙うなら、個数を増やすより先にまずサイズ上限を疑うとよい
- **光源の“ハロー”を広い指数(`pow(d, 5)`程度)で作ると、画面の1/4近くを覆う巨大な
  白飛びになった。** 宇宙空間に浮かぶ小さな光点にしたいなら、コアは`pow(d, 400)`前後、
  すぐ周りの締まったハローも`pow(d, 40)`前後まで指数を上げて、思ったよりずっと
  タイトに絞る必要がある
- フレネル反射だけ(指数を高くしすぎる)と、カメラ正面付近のファセットが屈折色(暗い
  背景グラデーション)一色になり単調なグレーの塊に見えた。プロトタイプ#3で効いた
  「冷たい主光源のdirectional lift + 暖色リム光」の2項をそのまま流用したところ、
  同じように色温度のコントラストが生まれて「艶消しの石」感が大きく減った
- 色収差は3回の`refract()`(R/G/Bで屈折率を`±0.014〜0.016`ずらす)だけで十分見える。
  背景に完全な単色ではなく多少コントラストのある環境(光源・グラデーション)を
  用意しておかないと、屈折方向をずらしても色の差がほとんど出ない点に注意
- 破片の頂点ジッターは `aBasePos`(=各頂点自身の位置ベクトル)方向にスケールするだけ
  なので、各頂点が独立にジッターしても破片の閉じたメッシュ構造が破綻しない
  (自己交差や穴が開かない)。頂点法線を持たず`dFdx`/`dFdy`で法線を求めているため、
  ジッターで三角形の向きが多少暴れてもフラットシェーディングは常に正しく追従する
- **screen-space refraction用のcaptureテクスチャは、シェーダーの分岐でサンプルしなくても
  ハマる罠がある。** captureパス(1パス目)の描画先フレームバッファに紐づくテクスチャが、
  そのユニットに前フレームの2パス目からバインドされたまま残っていると、GLSL側で
  `if (uCaptureMode) ...` により参照していなくても `GL_INVALID_OPERATION: Feedback loop
  formed between Framebuffer and active Texture` で描画そのものを拒否される。captureパスの
  直前に必ず `gl.bindTexture(gl.TEXTURE_2D, null)` でそのユニットを明示的に空にする必要がある
- captureテクスチャはRGBA8(プロトタイプ#2の教訓どおりfloatは避ける)なので、HDRの
  ハイライト(スパークルなど1.0を超える値)をそのまま書くとクランプされてしまう。
  Reinhard(`col/(col+1)`)だけを先にかけて0〜1に圧縮してから保存し、サンプル時に
  逆関数(`enc/(1-enc)`)で線形空間に戻すことで、ガンマ補正は最終出力の1回だけで済ませつつ
  HDRのハイライトも(トーンマップの分だけ圧縮されるが)ちゃんと持ち越せる
- 屈折方向のサンプルが実際にcaptureテクスチャを参照できているかは、目視だけでは
  判断しづらかった。`sampleBehind()`の戻り値を一時的に「実テクスチャ使用=赤/画面外
  フォールバック=緑/カメラ後方=青」に置き換えるデバッグ出力にしたところ、画面のほぼ
  全域が赤(=実際にcaptureテクスチャを覗けている)になり、狙い通り動いていることを確認できた
- **最初は反射率(フレネルのF0・指数)と屈折率(色収差用の`refract()`)が別々の
  決め打ち値で、互いに無関係だった。** 見た目だけ調整していると物理的な一貫性が
  失われやすいので、両方とも同じ屈折率`n_d`(破片ごとにクリスタルガラスの実測レンジで
  ランダム化)から導出するように直した。反射率はSchlick近似、R/G/Bの屈折率差は
  アッベ数の定義式から逆算している。指数を物理的に正しい5に戻しても(以前は
  正面付近が単調グレーに見えるのを避けるため3.2に下げていた)、screen-space
  refractionで屈折側に本物の背後の破片が映るようになった今は、屈折色自体に
  ディテールがあるため単調に見えず問題なかった
