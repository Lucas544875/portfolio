# メタボール文字 — プロトタイプ(ガラスの水滴で「川」を書く)

`portfolio-concept.md` の「結露をなぞると、名前の漢字が一画ずつ現れる」演出を、
2Dの拭き取りシェーダーではなく **3D空間に浮かぶガラスの水滴(球)が smooth-min で
液体的に融合しながら、正しい筆順で一画ずつ文字の形に凝集していく** ——という
フルレイマーチのガラス表現に置き換えた単体プロトタイプ。

[`../prototype/`](../prototype/)(KanjiVGなぞり)の "筆順に沿ってパスをなぞる" 考え方と、
[`../prototype-shards/`](../prototype-shards/)(ガラスの破片)のガラス質感(フレネル
反射/屈折 + 分散による色収差)を組み合わせている。フレームワーク無し・ビルド不要の
独立した WebGL2 アプリ。

## 実行方法

```sh
cd prototype-metaball-text
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## できていること

- **文字「川」を、筆順つきの簡略スケルトンからコード内データとして定義**している
  (フォント精度ではない)。各画は正規化平面 `[-1,1]`(x:右+, y:上+)上の
  2次ベジェ1本。直線的な画は制御点を中点付近に置いた退化ベジェ。
  `STROKES` 配列の並び順がそのまま筆順(**左 → 中央 → 右**)。
- 各画を **弧長で等間隔サンプル**(`STROKE_SAMPLE_SPACING`)して点列にし、
  各点に「水滴(球)」を1個ずつ置く。隣り合う球は `半径 > サンプル間隔` になるよう
  配置してあり、バラバラの玉ではなく1本の連続した画のブロブになる。
  水滴は共有のガラス板平面(z≒0)上に、ごく僅かに前後(`DEPTH_JITTER`)させて置く。
- **smooth-min はIQの多項式版**(`h = clamp(0.5 + 0.5*(b-a)/k, 0, 1); mix(b,a,h) - k*h*(1-h)`)。
  exp版より安価で、`k`(`SMIN_K`)がそのまま距離の単位=ブレンド幅になり直感的なので採用。
  全アクティブ水滴を横断して smooth-min するので、同じ画の中でも、
  後続の画が近ければ画同士でも滑らかに融合する(「川」では画が離れているので実際には
  画ごとに独立したブロブになる — 後述)。
- **単一の `progress`(0→1)uniform** が「名前をどこまでなぞったか」を表す。
  `head = progress * (水滴数 + GROW_SPAN)` を "書きヘッド" とし、
  水滴 `i` の出現度 = `clamp((head - i)/GROW_SPAN, 0, 1)` を smoothstep で
  滑らかにして半径に掛ける。**筆順どおり**に水滴を並べてあるので、
  `progress` を上げると 左画 → 中央画 → 右画 の順に、各画は上端から下端へ育つ。
- **未出現の水滴の完全ゲート**: 出現度が 0 のとき実効半径はちょうど 0 になり、
  シェーダーの `if (dp.w <= 0.0) continue;` で SDF から**完全に除外**される。
  smooth-min に一切寄与しないので、まだ現れていない水滴が場に幽霊のような膨らみを
  作ることはない(検証は後述)。
- ガラス質感は `../prototype-shards/` を踏襲: フレネル(Schlick)で反射/屈折を配分し、
  アッベ数の定義から R/G/B ごとに屈折率をずらして実際に3回 `refract()` することで
  色収差(分散)を出す。背景・スタジオ光源風の光点・きらめき・トーンマップ関数は
  そのまま流用して世界観を揃えている。屈折色にはうっすら水色の吸収(`DENSITY_TINT`)を
  掛けて「水/ガラスの厚み」の気配を添える。
- 読み込み時は `progress` がゆっくり自動進行して「川」を書き切り、少し溜めてから
  ループする(他プロトタイプの自動再生の作法)。**ユーザーがドラッグ(横方向)/
  スクロールした瞬間に手動スクラブへハンドオフ**する(`../prototype-shards/` の
  「自動再生 → ドラッグでハンドオフ」と同じ引き継ぎ方。ただし本作は操作対象が
  カメラの周回ではなく `progress` のスクラブ)。カメラ自体はごく僅かに首を振るだけ。
- モバイル(`matchMedia("(pointer: coarse)")`)では内部解像度を自動的に落とす。

## なぜ「川」か(文字の選定理由)

- サイトの **水・ガラス・結露のモチーフに合う**(川 = river)。
- **3画すべてが空間的に分離した縦画**で、メタボールにしたとき「玉の連なり」ではなく
  「読める文字」になりやすい。画が交差する字(例: 水)は交点で玉が団子になり
  可読性が落ちるリスクが高い。
- **筆順が明快で自信を持てる**: 左の画 → 中央の画 → 右の画、各画は上から下へ。これは
  日本の学校教育でも一般的な標準筆順。"水" は 2・3画目の形状の記憶が曖昧で、正しい
  筆順を断言しづらかったため今回は見送った(タスクの「不確かなら明記せよ」に従い、
  確信の持てる字を選んだ)。

## 水滴数(パフォーマンス予算)

レイマーチは step ごとに全水滴と smooth-min するため、水滴1個 = 全ピクセル・全 step の
コスト増になる。`../prototype-shards/` の破片数(100〜150、ただしあちらはラスタライズ)と
同オーダーに収めることを狙い、`MAX_DROPLETS = 120` を上限にした。

現在の実測値(`STROKE_SAMPLE_SPACING = 0.11`):

| 画 | 弧長 | 水滴数 |
|---|---|---|
| 1画目(左) | 1.60 | 16 |
| 2画目(中央) | 1.00 | 10 |
| 3画目(右) | 1.72 | 17 |
| **合計** | | **43** |

**43個**で上限にも予算にも十分余裕がある。もし別の(画数の多い)字に差し替えて可読性の
ために点が増える場合は、**水滴数の上限を超えるのではなく、画ごとに
`STROKE_SAMPLE_SPACING` を広げてサンプル密度を下げる**こと(上限に達すると
コンソールに警告を出して打ち切る安全装置あり)。

## 数値で確かめたこと(ブラウザ無しでの検証)

環境にヘッドレスブラウザが無いため、描画結果は目視できない。代わりに JS のジオメトリ
計算を切り出して数値で確認した:

- **画内の連続性**: 隣り合う水滴の中心間隔の最大値は約 `0.111`。球の直径
  `2 * DROPLET_RADIUS = 0.17` がこれを上回るので、smooth-min 以前に球の union だけで
  既に穴なく繋がる。
- **筆順の自己整合**: 各画の弧長サンプルは `t=0`(=ベジェの `p0`)から始まる。
  全画の `p0` を画の**上端**(y が最大側)に置いてあるので、水滴列は常に
  上 → 下 の描き順に並ぶ。例えば1画目(左)は `p0=(-0.52,0.72)`(上)から、
  中点 `t=0.5` で約 `(-0.645,-0.06)`、`p1=(-0.74,-0.86)`(下)へと、y が単調に
  減少しつつ x も単調に左へ寄る。**どの画も、自分自身の水滴だけで正しい形になり、
  後続の画の水滴が先に存在している必要はない。**
- **画同士が誤って融合しないこと**: 異なる画に属する水滴の中心間距離の最小値は
  約 `0.57`。ブレンドが効く範囲 `2*DROPLET_RADIUS + SMIN_K ≒ 0.29` を大きく上回るので、
  「川」の3画は smooth-min しても互いに独立したブロブのままになる(= 3本の縦画として
  読める)。
- **未出現ゲートに幽霊の膨らみが出ないこと**: 未出現の水滴は実効半径ちょうど 0 で、
  シェーダー側は `if (dp.w <= 0.0) continue;` によりその水滴を **smooth-min に一切
  渡さない**。SDF の初期値は `1e5`(シーン半径〜2・カメラ距離〜3 に対して桁違いに
  大きい番兵)で、最初の1個目の smooth-min で `h` がちょうど 0 にクランプされ、
  補正項 `k*h*(1-h)` も 0 になるため、番兵由来の膨らみも出ない。
- `node --check main.js` は通過(構文エラーなし)。

## 分かったこと・チューニングのメモ

- **`SMIN_K` は「どれくらい液体的か / ボコボコか」を決める最重要ノブ。** 大きすぎると
  画が痩せて溶けすぎ、小さすぎると水滴が団子状に分かれて見える。`0.12` は
  半径 `0.085`・間隔 `0.11` に対する当てずっぽうの初期値で、実見での再調整が要る
  (下記「視覚的に未検証な点」参照)。
- 出現の育ち幅 `GROW_SPAN` を広げると、書きヘッド付近で複数の水滴が同時にじわっと
  育ち、より「にじむように現れる」印象になる。狭めると1個ずつパチッと現れる。
- カメラは意図的にほぼ正面固定(ごく僅かな首振りのみ)。操作(ドラッグ/ホイール)は
  カメラ周回ではなく `progress` のスクラブに割り当てているので、文字の可読性が
  常に保たれる。

## 視覚的に未検証な点(実ブラウザでの調整が要る箇所)

**この環境では一切レンダリングを目視できていない。** 数式・座標は上記のとおり
数値検証したが、下記は "見た目" の問題なので人間の目での確認・調整がほぼ確実に必要:

1. **`SMIN_K`(smooth-min のブレンド幅)は当てずっぽう。** `0.12` が「とろけた液体」に
   見えるか「ただ太らせた棒」に見えるか、あるいは玉の継ぎ目が残るかは未確認。
   半径・サンプル間隔とセットで見ながら詰める必要がある。
2. **サンプル密度と可読性は未検証。** `STROKE_SAMPLE_SPACING = 0.11` / 半径 `0.085` で
   「川」が読める太さ・連続性になるか、画が太すぎ/細すぎないかは目視待ち。
   特に中央画が短く見えすぎないか。
3. **筆順は標準的と考えているが、KanjiVG など一次資料との突き合わせは未実施。**
   「川」は 左→中央→右・各画上から下、と確信しているが、正式データでの裏取りはしていない。
4. **カメラの画角(`FOV`/`CAM_DIST`)と文字の画面占有率**は計算上「高さの約7割」に
   なる見積もりだが、実際に文字が画面に収まって主役に見えるかは未確認
   (縦横比・端の見切れ含む)。
5. **ガラスの光学の見え方**: 単一界面近似(屈折の先はプロシージャル背景のみで、
   `../prototype-shards/` のような "奥の水滴が透ける" スクリーンスペース屈折は未実装)。
   細い画の中で屈折・色収差・きらめきが破綻せず "水滴らしく" 見えるか、`IOR` /
   `ABBE_NUMBER` / `DENSITY_TINT` が妥当かは目視での調整が要る。
6. **`SURF_EPS` / `MAX_STEPS` の十分性**: smooth-min した薄い場を細い画で
   マーチするとき、輪郭にノイズやリング状のアーティファクトが出ないかは未確認。
   出たら `SURF_EPS` を下げる / step 数を増やす調整が要る。
