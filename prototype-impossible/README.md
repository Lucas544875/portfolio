# 不可能な結晶 — プロトタイプ #5(depth-swap トリックによる錯視)

視点をひとつに固定したときだけ「1本のつながったガラスの結晶」に見え、少しでも動かすと
実は前後に離れた2つの別ピースだったと割れて見える——杉原厚吉の「不可能立体/ありえない
オブジェ」と同じ系統の錯視を、レイマーチの depth-swap トリックで作った単体プロトタイプ。

元ネタ: [`trick.frag`](https://hausdorff-dimension.netlify.app/trick.frag) の
「通常レイマーチで当てた面を、視線方向へ踏み越して別のより単純な距離関数で再マーチし、
内部を貫通して"奥の面"を掴んで置き換える」という仕組みを、独自の形状・インタラクションで
作り直したもの。[`../prototype/`](../prototype/)(KanjiVGなぞり)・
[`../prototype-rain-window/`](../prototype-rain-window/)(雨の窓)・
[`../prototype-shards/`](../prototype-shards/)(ガラスの破片)とは独立した、完全に別の WebGL2 アプリ。

フレームワーク無し、ビルド不要。

## 実行方法

```sh
cd prototype-impossible
python3 -m http.server 8080
# → http://localhost:8080 をブラウザで開く
```

## できていること

- フルスクリーンのレイマーチ(1つの SDF シーンをピクセルごとに描く。破片=#4の
  ラスタライズとは別系統で、#3(結晶)の直系)。ただし普通のレイマーチに
  **depth-swap トリック**を1段足している:
  1. `distanceFunction(p)`(結晶2ピース + 床)で通常のレイマーチをして「一番手前の面」に
     当てる = `nearHit`
  2. その点から視線方向へ固定量 `TRICK_OVERSTEP` だけ踏み越し、今度は `depthFunction(p)`
     (**床を含まない、結晶2ピースだけ**)に対してもう一度マーチする。距離が負
     (=ピースの内部)の間は sphere-trace せず固定量ずつ前進して**内部を貫通**し、
     次に出会う面(=奥のピースの面)を掴む
  3. その「奥の面」で `nearHit` を置き換えて陰影計算する
- **錯視の形状設計:** 2本のガラス角柱(rounded box)を、スイートスポットの視線(=-Z)に
  沿って前後に大きく(深さ 2.0)離し、かつ横(X)にずらして置いている。
  - ピースA(手前 +Z 側): `x∈[-2.3, 0.7]`
  - ピースB(奥 -Z 側): `x∈[-0.7, 2.3]`
  - スイートスポット(=Z 軸上からまっすぐ見る)からは A と B の投影が中央 `x∈[-0.7,0.7]` で
    重なり、合わせると `x∈[-2.3,2.3]` の**1本の長い結晶棒**に見える。重なり領域では手前Aに
    当たった光線をトリックが貫通させて奥Bの面に差し替えるので、A→B の継ぎ目が段差なく
    「奥へつながった1本」として読める(本当は前後 2.0 離れた別物なのに)
- **崩し方(インタラクション):** トリックの効き `uTrickAmount`(0〜1)を、現在のカメラ前方
  ベクトルとスイートスポット方向の内積の `smoothstep` で連続ブレンドする。スイートスポットに
  近いほど 1.0(全開=つながって見える)、離れるほど 0.0(素の `nearHit`=手前Aの面をそのまま
  描く=A と B が前後・左右にズレた別々の2ピースだと露呈)。ブレンドは
  **「ヒット位置・法線を mix してから1回だけ陰影計算」する真のクロスフェード**なので、
  視点を動かすと錯視が"割れて"いく過程がなめらかに見える(=このプロトタイプの主眼)
- ガラスの陰影は #4 と同じ考え方の簡略版:屈折率 `IOR` から Schlick 近似でフレネル反射率を
  導出し、R/G/B をアッベ数 `ABBE` から求めた別々の屈折率で `refract()` して色収差を出す。
  屈折先はプロシージャルな環境(背景)だけを覗く(単一オブジェクトなので #4 のような
  screen-space refraction の2パスは省略した)
- 参考デモに合わせ、**床は `distanceFunction` にだけ入れ `depthFunction`(=トリックと影)には
  入れない**。床には結晶のソフトシャドウが落ちるので、視点を動かして錯視が割れると
  「影も2つに分かれている」ことが接地の手掛かりになり、2ピースである事実を後押しする
- カメラは #1〜#4 と同じ「自動再生 → ドラッグでハンドオフ」。ただし錯視が主役なので、
  ロード直後はまず**スイートスポットで静止**(`LINGER_SECONDS`)して"完成した錯視"を見せてから、
  ゆっくり視点をドリフトさせて崩れ始める。ユーザーがドラッグした瞬間に自由オービットへ引き継ぐ
- モバイル(`matchMedia("(pointer: coarse)")`)では内部解像度とレイマーチ/トリック/影の
  反復回数を自動的に減らす
- 人間が目視後にチューニングする値(スイートスポット方向・トリックのブレンド角度範囲・
  2ピースの位置とサイズ・overstep 量など)はすべて `main.js` 冒頭の `CONFIG` に名前付き定数として
  コメント付きで集約してある

## 分かったこと・チューニングのメモ

- **この手の錯視は「近オルソ」(遠くから狭い画角)が正しい領域。** 最初カメラを近く
  (`CAM_DIST` 5.2)に置いていたが、前後に 2.0 離れた2ピースは遠近で見かけの太さが
  20〜30%も違い、継ぎ目で棒が細くなって錯視が成立しない。カメラを引いて(`CAM_DIST` 9.0)
  `FOV` を上げ(2.9)、投影を近オルソにすることで見かけの太さの差を圧縮している。
  それでも残差(下記「未検証」参照)はある
- **overstep 量(`TRICK_OVERSTEP`)は錯視の要でありながら壊れやすい。** 大きすぎると最初の
  踏み越しで奥ピースBの内部に着地してそのまま貫通し、Bを飛び越えて何も掴めなくなる。
  本配置では A 前面(z≈1.41)〜B 前面(z≈-0.59)が 2.0 なので `OVERSTEP < 2.0` を守れば
  最初の踏み越しは必ず手前のギャップに落ち、そこから sphere-trace で確実にBの面へ寄る。
  0.6 に設定
- トリックの再マーチは**必ず反復回数(`TRICK_STEPS`)と距離(`TRICK_MAX_DIST`)でキャップ**し、
  奥に面が無ければ `found=false` で `nearHit` にフォールバックする。これにより
  「A しか通らない光線」(奥にBが無い)や「Bを直接見ている光線」(Bの先には何も無い)でも
  無限ループや暴走が起きず、素直に手前の面が出る
- 法線がゼロベクトルになって NaN が出ないよう、位置・法線の mix 後は
  `length(mixedN) > 1e-4` を確認してから正規化している(手前と奥の法線がほぼ逆向きに
  なるケースの保険。実際は A・B とも前面が +Z を向くので通常は起きない)
- 全反射時の `refract()` は 0 ベクトルを返すが、本作は常に空気→ガラス(eta<1)なので
  理論上は起きない。数値誤差の保険として反射色フォールバックだけ残している

## 視覚的に未検証な点(ブラウザでの目視後に調整が要りそうな箇所)

**この環境ではブラウザ描画を目視できないため、以下は数式・シミュレーションで詰めただけで、
"実際にそう見えるか"は未検証。** 特にリスクの高い順:

1. **【最重要】継ぎ目の「太さの一致」= 遠近補正。** A(手前 dist≈8)と B(奥 dist≈10)は
   カメラ距離が違うので、同じ断面 `0.34` でも B の見かけの太さは A の約 0.8 倍になり、
   継ぎ目で棒がわずかに細くなる可能性が高い。`CAM_DIST`/`FOV` の近オルソ化で緩和しては
   いるが残差はあるはず。**目視で継ぎ目が細るなら、B のサイズを A の
   `dist_B / dist_A` 倍(≈1.25 倍。`PIECE_B_HALF` の y,z を `0.34 → 0.42` 前後)に
   拡大して補正する**とよい(割れたときは B が少し太い別ピースに見えるが、それは
   露呈演出として許容範囲)
2. **継ぎ目(x≈±0.7)で A の前面と B の前面の陰影が本当に連続に見えるか。** 両方 +Z を向く
   ガラス面で同じ環境を映すので近いはずだが、深さ 2.0 差があるため反射・屈折で拾う背景の
   ハイライトがずれ、継ぎ目で色/明るさが飛ぶ可能性がある。飛ぶなら 2ピースの深さ分離
   (`PIECE_*_CENTER` の z)を縮める、または `background()` のハイライトを緩くする
3. **トリックのブレンド角度範囲(`TRICK_COS_INNER`/`OUTER` = 6°/22°)は完全な推定値。**
   割れ始めるのが早すぎる/遅すぎるなら角度を広げる/狭める。理想は「静止して見とれる余裕は
   あるが、少し動かすとハッと割れる」バランス。目視前提のマジックナンバー
4. **スイートスポット方向(yaw=pitch=0 の真 -Z)は代数的に選んだだけ。** 2ピースを -Z に沿って
   並べた設計上ここが唯一の正解のはずだが、"継ぎ目が実際につながって見える"のがこの向きで
   合っているかは要確認。より 3/4 アングルの見栄えのするスイートスポットにしたい場合は、
   `SWEET_YAW`/`SWEET_PITCH` と `PIECE_*` 配置を**一緒に**作り直す必要がある(片方だけ変えると
   投影の一致が崩れる)
5. **overstep が実際に「手前Aの面 → 奥Bの面」を狙い通り差し替えているか。** シミュレーション上は
   重なり領域の光線が B 前面(z≈-0.59)に収束し、A のみ/B のみの光線はフォールバックする
   ことを確認済みだが、丸め角(`PIECE_ROUND`)や法線 mix の影響で継ぎ目付近が濁る可能性がある。
   デバッグ時は `uTrickAmount` を 1.0 固定にし、トリックのヒット位置(near=赤/far=緑/
   フォールバック=青)を色分け出力すると効き具合を確認しやすい
6. 棒の**長さ・太さ・重なり幅の見た目のバランス**(`PIECE_*_HALF`、中心の x)は未調整。
   重なりが狭すぎるとトリックが効く帯が細く、広すぎると"つながり"の説得力が下がる
7. ガラスの色味・地色・スペキュラ強度・`GLASS_TINT`・フォグ量は #3/#4 の値を流用した当て推量。
   結晶が暗く沈む/白飛びするなら要調整

## 次のステップ

- 錯視が割れたときに「2ピースだった」ことがもっと明快に伝わる演出(割れた瞬間だけ
  ピース間にラインやラベルを出す等)
- 形状を単純な2本の角柱から、より結晶らしい/より"ありえない"コーナー(ペンローズ三角形の
  一角など)へ発展させる。その場合トリックの overstep とピース配置の再設計が要る
