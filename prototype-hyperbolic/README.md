# 定曲率空間のレイマーチ — プロトタイプ #11

「無限に広い床 + 等間隔に並ぶ円柱 + フラクタルな木」を、曲率 K が連続的に
変えられる空間の中でレイマーチして描画し、画面を左右分割して**同じ格子を
常にユークリッド空間で描いた絵**と並べる。右ペインは K<0(双曲空間)・
K=0(ユークリッド空間)・K>0(球面)をスライダーでシームレスに行き来できる
——3つの幾何は別々の実装ではなく、曲率で連続的にパラメータ化した1本の
数式群(一般化三角関数 cosK/sinK)から導かれている。フレームワーク無し、
ビルド不要。

## 実行方法

```sh
cd prototype-hyperbolic
python3 -m http.server 8000
# → http://localhost:8000 をブラウザで開く
```

カメラは木を中心に自動で周回する。ドラッグで視点操作(yaw/pitch)、
ホイールで周回速度を加速。左がユークリッド版(常に K=0)、右が曲率
スライダーで操作できる定曲率空間(既定値 K=-1、双曲空間)。

## できていること

- 曲率 K を連続パラメータとして持つ、定曲率3次元空間の統一実装。
  一般化三角関数 `cosK`/`sinK`(K<0 で cosh/sinh, K>0 で cos/sin, K→0 で
  1/x に一致)と、それに対応する一般化内積 `gdot` を使って、床・柱・木・
  カメラの全ての式を1本の K パラメータ付き公式で書き、K の符号で分岐する
  のは「その式が数値的に安定な範囲」だけに留めている(詳細は後述「曲率の
  パラメータ化」節)。
- 双曲面モデル(hyperboloid model)の一般化による定曲率3次元空間の実装。
  床(全測地的超平面)と、格子状に並べた円柱(鉛直な測地線からの管)を、
  距離を返す SDF として厳密に定式化し、直線ではなく測地線に沿って
  スフィアトレースする。
- 同じ格子インデックス・同じ操作入力(周回角 theta、見回し角 yaw/pitch)を、
  ユークリッド版・定曲率版それぞれの「周回」「回転」の公式に通すだけの
  完全に並行なカメラ実装。見え方の違いが純粋に曲率の差から来るようにした。
- 左右分割レンダリング(`gl.viewport` を2回呼ぶだけで、1本の `<canvas>` に
  2つのシェーダープログラムを描く)。
- 周回の中心に、細い円柱の再帰的コピーからなるフラクタルな木を1本配置。
  枝分かれの規則(角度・本数・減衰率・再帰段数)は完全に共通の JS コードで
  1回だけ定義し、ユークリッド版・定曲率版それぞれの「移動フレームを1歩分
  前進させる」処理だけを差し替えて具体化する(詳細は後述「フラクタルな木」
  節)。床・柱に続く3つ目の比較材料で、カメラはこの木を見ながらその周りを
  巡る(詳細は後述「周回カメラ」節)。木は K が変わるたびに JS 側で再生成し
  uniform 配列として渡す(const 配列に焼き込むと曲率を変えるたびにシェー
  ダー再コンパイルが必要になり、スライダーの応答が重くなるため)。

このプロトタイプは、素直な実装がことごとく「一見動いているように見えて
実は何も描画していない」形で壊れる、という双曲幾何特有の罠の連続だった。
以下、数学の要点とセットで、踏んだ罠を時系列で書き残す。

## 双曲面モデルの要点

H³ を R^{1,3}(ミンコフスキー空間、符号 `-+++`)の中の双曲面
`⟨p,p⟩ = -1, p.t > 0` として実装している。4成分は `(t, x, y, z)` の順で
GLSL の `vec4` に `(.x=t, .y=x, .z=y(上方向), .w=z(横))` として詰める。
ミンコフスキー内積は `mdot(a,b) = -a.x*b.x + a.y*b.y + a.z*b.z + a.w*b.w`。

### 床(全測地的超平面)までの距離

法線 `n=(0,0,1,0)` を持つ超平面までの符号付き距離は

```
floorSDF(p) = asinh(⟨p, n⟩) = asinh(p.z)
```

球面上の大円までの角距離が `arcsin` になるのと同じ形の恒等式が、双曲版
(ローレンツ内積)では `arcsinh` になる。

### 円柱(鉛直な測地線からの管)までの距離

点 `(x0, 0, z0)` を通る鉛直な測地線 L は、時間的単位ベクトル
`a=(t0,x0,0,z0)`(`t0=sqrt(1+x0²+z0²)`)と空間的単位ベクトル `b=(0,0,1,0)`
が張る2平面と双曲面の交わりとして定義できる(`⟨a,a⟩=-1, ⟨b,b⟩=1,
⟨a,b⟩=0` を満たす)。このとき点 p から L までの距離は、

```
cosh(d(p,L)) = sqrt(⟨p,a⟩² − ⟨p,b⟩²)
```

という閉じた式になる。導出は「p を span{a,b} 上の成分 `p_parallel` と
その直交補空間の成分 `p_perp` に分解し、双曲面の制約
`⟨p,p⟩=-1=⟨p_parallel,p_parallel⟩+⟨p_perp,p_perp⟩` と
`⟨p_parallel,p_parallel⟩ = -⟨p,a⟩²+⟨p,b⟩²` を組み合わせる」というもので、
直交補空間の基底を明示的に求める必要がない(実装上ありがたい)。円柱の
SDF はこれから半径を引くだけ: `cylSDF(p) = acosh(cosh(d)) - CYL_RADIUS`。

### 測地線に沿ったスフィアトレース

直線ではなく測地線に沿って進める。SDF が返した"双曲距離" Δ に対し、

```
p' = p·cosh(Δ) + d·sinh(Δ)
d' = p·sinh(Δ) + d·cosh(Δ)
```

(`p`=現在位置, `d`=現在の進行方向、どちらも `⟨p,p⟩=-1, ⟨d,d⟩=1, ⟨p,d⟩=0`
を満たす)。球面上の測地線が `cos/sin` で進むのとまったく同じ形で、双曲版
は `cosh/sinh` になる。`d'` は毎ステップ `p'` への接空間へ再射影
(`v_tan = v + ⟨v,P⟩P`, `⟨P,P⟩=-1` を利用)して正規化し、数値誤差の蓄積を
防いでいる。

## フラクタルな木(床・柱に続く3つ目の比較材料)

通路の中心に、細い円柱の再帰的コピーからなるフラクタルな木を1本立てている。
「枝分かれの規則」自体はユークリッド・双曲どちらでも意味を持つ抽象概念
(角度・本数・長さ/太さの減衰率・再帰段数)なので、生成コードを2つの幾何で
二重に書きたくない。そこで `buildTree()` に再帰そのものを1回だけ実装し、
「フレーム(位置 + 3方向の正規直交基底)を1歩分だけ前進させる」処理
(`move()`)と「回転」(`rotateFrame()`)という2つの操作だけを adapter
として差し替える設計にした。

- **回転**は驚くほど単純に共通化できる。フレームの3方向はどちらの幾何でも
  「正規直交」(ユークリッドなら通常の内積、双曲ならミンコフスキー内積の
  意味で)なので、2方向を角度 θ で混ぜる標準的な回転式
  `a' = a·cosθ + b·sinθ, b' = -a·sinθ + b·cosθ` が、成分の中身がベクトル3
  次元でも4次元でも**そのまま**成立する。
- **前進**だけが幾何ごとに違う。ユークリッドは `P' = P + F·length` という
  単純加算。双曲はカメラの前進と全く同じ式
  `P' = P·cosh(length) + F·sinh(length)`(測地線に沿った輸送)で、しかも
  移動方向に直交する残り2方向(S1, S2)は「罠3」で確認したのと同じ理屈で
  不変のまま運べる。

双曲版の木の根元は、原点から双曲距離 `s` だけ x軸(z=0, 床の上)を進んだ点
に置いている。「上方向」の初期値には `F=(0,0,1,0)` を使っているが、これは
床の SDF (`floorSDF(p)=asinh(p.z)`) の法線と全く同じベクトルで、
`⟨P,(0,0,1,0)⟩=0` が床の上のどの点 `P=(t0,x0,0,z0)` でも成り立つため、
「その場でのローカルな上方向」を測地線輸送で求め直す必要がない。

木の枝(有限長の測地線分)を管の形にするには、柱で使った「無限直線までの
距離」の式を線分用に拡張する必要がある。枝の始点 A・始点での接ベクトル
D・弧長 L が分かっているとき、無限測地線上の最近点のパラメータ `s*` は
柱の距離公式の導出と全く同じ分解 `P = c_A·A + c_D·D + 直交成分`
(`c_A=-⟨P,A⟩, c_D=⟨P,D⟩`)を使って `s* = asinh(c_D / k)`
(`k=sqrt(c_A²-c_D²)`)と閉じた形で求まる。これを `[0, L]` にクランプして
`A·cosh(s)+D·sinh(s)` を計算すれば、線分上の最近点までの真の双曲距離が
得られる(ユークリッドの capsule SDF の双曲版)。

## 罠1: 格子を素朴にチャート座標で等間隔にすると柱同士が重なる

最初は `x0=i·SPACING, z0=(j+0.5)·SPACING` という、双曲面モデルのチャート
座標をそのまま整数倍しただけの格子を使っていた。ところがユーザーから
「双曲空間において円柱同士が重なってしまっている、これは変じゃないか」と
指摘され、実際に検証すると本当に重なっていた。

原因はチャート座標と真の双曲距離の関係にある。原点から `(x0,0)` までの
真の双曲距離は `acosh(sqrt(1+x0²))` で、これは `x0` に対して**対数的**に
しか伸びない(`x0` が大きいところでは `d'(x0) → 0`)。つまりチャート座標
上で等間隔(`x0=i·S`)に並べても、原点から離れた柱ほど**真の隣接距離は
どんどん縮んでいき**、半径 0.5・間隔 3.0 の設定でも `i=2`(x0=6)で
早くも隣接柱間の真の距離(0.70)が直径(1.0)を下回って重なった。

修正: 2つのローレンツブースト(x方向→z方向)の合成で格子点を作る方式に
変更した。

```
x0 = sinh(n·SPACING)
z0 = cosh(n·SPACING)·sinh(m·SPACING)
```

1径数部分群(ある一方向への平行移動)の軌道は、群の構造から
`d(Boost(s)·p, Boost(s+Δ)·p) = d(p, Boost(Δ)·p)` が**任意の p で** 成り立つ
(両辺に `Boost(s)⁻¹` を掛けるだけの恒等式)。つまり同じ行内(m固定、
n→n+1)の間隔は、その行が原点からどれだけ離れていても常にちょうど
SPACING になる。列方向(n固定、m→m+1)の間隔は軸(z=0)から離れるほど
「広がる」側に振れる(`cosh²(nS)(cosh(S)-1)+1` の形で `n` について単調
増加する)ため、近づいて重なる心配がない。SPACING=3.0 のような大きな値は
1セル進むだけで `cosh(3)≈10` と埋め込み座標が暴れて格子が大きく歪むため、
ユークリッド側とは独立に SPACING=0.9, CYL_RADIUS=0.3 まで縮めてある
(`EUC_SPACING`/`HYP_SPACING` を参照。同じ添字グリッドを異なる幾何で解釈
する、という趣旨は変わらないが、具体的な間隔の数値までは両者で揃えて
いない)。

## 罠2: `gl_FragCoord` はウィンドウ座標

左右分割で右ペインを `gl.viewport(halfW, 0, ...)` に描くとき、
`gl_FragCoord.xy` はビューポート内の相対座標ではなく常にウィンドウ全体の
座標系で返る。これに気づかず
`uv = (gl_FragCoord.xy - 0.5*uResolution)/uResolution.y` をそのまま書いた
ところ、右ペインのレイ方向が丸ごとズレて画面全体が空振りした。
`uViewportX` uniform でオフセットを補正している。

## 罠3: 前進経路を測地線にすると床から飛んでいく

前進を「測地線に沿った直進輸送」で実装したところ、床と平行に出発した
測地線が床から指数的に遠ざかっていき、わずか数ステップでカメラが視野の
外(はるか上空)まで飛んで行ってしまった。双曲空間では「ある平面に平行
かつ等距離であり続ける道」は直線(測地線)にならない、という平行線公準が
破れる核心そのものだが、歩行カメラとしては使えない。2つのローレンツ
ブースト(高さ方向→前進方向の順)を原点の正規直交フレームへ合成適用する
ことで、床からの高さが恒等的に一定になる経路(測地線ではなく等距離曲線)
に置き換えた。ブーストは等長写像なので、フレーム全体に同じ行列を掛ける
だけで正規直交性が自動的に保たれる(詳細は `hyperbolicCamera()` の
コメント)。

## 罠4: 前進すると格子が急速に真横・背後へ回り込む

罠1・罠3を修正したあとも、右ペインには床しか映らず柱が一切見えない、
という状態が続いた。`hit` フラグ自体は正しく true になっていた(床には
ちゃんと当たっている)ため、シェーディング側のバグを疑って `calcNormal`/
`shade`/合成処理を1つずつバイパスして調べたが、変わらず柱には一度も
当たらないことが分かった。

最終的に Node.js へ距離計算を移植し、「レイが実際に到達しうる最寄りの
柱までの距離」を全方位・広い探索範囲でしらみつぶしに調べたところ、
カメラ位置から見て**どの方向を向いても柱の軸までの距離が半径を割り込ま
ない**ことが判明した。さらに、既知の柱へ数学的に正確な方向(2点を結ぶ
測地線の初期接ベクトル)を計算して直接レイを飛ばすテストを行うと、狙って
撃てば柱にちゃんと当たることも確認できた。つまり幾何としては柱は
到達可能だが、**「1マス先」のつもりの柱が、カメラの通常の視野角からは
とうに外れた方向にある**ことが根本原因だった。

原因はチャート座標と真の距離の対応が対数的である(罠1と同じ根)ことに
加えて、もう一段深い。カメラの前進距離 `t` は定義上そのまま原点からの
真の双曲距離だが、格子は「原点からのチャート座標」を基準に作られている
ため、`t` が線形に増えるとカメラは「1マス先の柱」の実際の位置(原点から
見て対数的にしか離れていない)をあっという間に追い越してしまう。しかも
柱の「横方向のオフセット」は「前進」と可換な変換ではない(ローレンツ
ブーストは非可換)ため、直前まで正面付近にあった柱が、前進するにつれて
急速に真横〜背後へ回り込んで見える。実測(高さ0.22, SPACING=0.9)では:

| t   | 「1マス先」の柱が見える角度(正面からの偏角) |
|-----|-----------------------------------------------|
| 0   | 約33°                                          |
| 0.3 | 約49°                                          |
| 0.7 | 約54°                                          |
| 1.0 | 約71°                                          |
| 1.3 | 約91°(視野の真横〜背後)                       |

カメラの高さを変えてもこの角度変化はほぼ同じで(=平行線角のような
「高さ依存の」現象ではなかった)、格子をチャート座標式・ブースト合成式の
どちらにしても同様に起きた。一方で「横オフセット無し(z0=0、カメラ自身の
測地線上の点)」を対象にすると、t に依らず視野角は安定していた。つまり
**横方向のオフセットを持つ対象は、観測者が前進するにつれて視野内での
角度が安定しない**という、非可換な等長変換群に由来する現象だと考えている
(ローレンツブーストは一般に非可換であり、「前進」と「横移動」を異なる
基準点(原点)からの独立した変換として合成すると、移動する観測者から見た
相対角度が保たれない)。

最初の対策は、前進の上限を大きく取らないこと(`T_MAX=0.6`)だった。この
範囲なら「1マス先」の柱はおおむね正面〜54°以内に収まり、広角レンズと
組み合わせて実用的に視界に収まってはいた。ただしこれは根本解決ではなく
症状を抑えただけで、「もっと遠くまで歩きたくなる」誘惑と常に隣り合わせの
ワークアラウンドだった。木を配置して「周遊する」動線に変えた際、この
制約と正面から向き合うことになり、次節の周回カメラという根本的な解決に
たどり着いた。

## 周回カメラ(罠4の根本的な解決)

カメラは今、木を中心とする円軌道(ユークリッド版は普通の円、双曲版は
木を中心とする双曲円)を周回する。罠4で見た問題――「前進すると対象までの
距離が伸び、しかも横オフセットは前進と可換でないため対象が急速に視野外
へ回り込む」――は、**対象からの距離を変えずに回り込む**ことで、そもそも
起こらなくなる。周回では原点からの距離が全く変化しないので、非可換性が
悪さをする余地がない。

実装は木の生成コード(`buildTree()`)と全く同じ「フレームを回転してから
測地線移動する」パターンの再利用になっている:

1. 木の根元 `C` を中心に、木の `initFrame()` と同じ接空間の基底
   `S1`(原点から半径方向), `S2`(横方向)を使い、周回角 θ で回転した
   単位接ベクトル `dir(θ) = S1·cosθ + S2·sinθ` を作る。
2. `dir(θ)` 方向へ測地線を `ORBIT_RADIUS` だけ進めた点が、床の上の軌道
   位置。木を見る(内向きの)方向は、その移動の外向き接ベクトルの符号を
   反転するだけで得られる。円に接する横方向 `tangentDir(θ) = -S1·sinθ +
   S2·cosθ` は移動方向と直交しているため、木の枝や罠3のカメラ前進と
   同じ理屈で不変のまま運ばれる。
3. そこから「上方向」`(0,0,1,0)`(床のどの点でも常に接空間内で単位・
   直交)へ `height` だけ測地線移動して、実際のカメラ高さにする。木を
   見る方向・円の接線方向はこの昇降とも直交しているため、やはり不変。
4. 最後に yaw→pitch の順に、接空間内の通常の回転として視線フレームへ回す
   (見回し操作は今までと同じ)。

前進距離だった `t` は周回角 `theta` になり、`T_MAX` のような上限は不要に
なった(2πごとに `% (2*Math.PI)` で巻き戻すだけ)。ホイールでの加速も
「前進速度への加算」から「周回角速度への加算」に変わっただけで、UI の
挙動自体は変えていない。

### 周回半径の衝突判定

半径を決める際にもう一つ罠を踏んだ。最初 `ORBIT_RADIUS_EUC=2.0`,
`ORBIT_RADIUS_HYP=0.9` で試したところ、ドラッグとホイールを組み合わせた
特定の操作の後に、片方のペインが単色でベタ塗りされる(床もレイマーチの
形跡も無い)という不可解な症状が出た。コンソールエラーは無く、`hit`
判定自体も一見正常。

Node.js へカメラ位置の計算をそのまま移植し、周回角 θ を 0〜2π で細かく
走査しながら `min(床SDF, 柱SDF, 木SDF)` の最小値(=周回軌道上でシーンの
どこかの面に最も接近する量)を調べたところ、**軌道の一部が柱の内部を
通っている**(クリアランスが負)ことが分かった。柱の間隔(ユークリッド
3.0・双曲 0.9)に対して周回半径が大きすぎ、木を中心にした円が途中で柱に
めり込んでいた。単色ベタ塗りは、カメラが柱の内側からレイマーチを始めた
ときの典型的な縮退症状(最初の一歩で `hit` が即座に成立し、法線もマテリ
アル判定も原点付近の妙な値になる)だったと考えられる。

対策は、周回半径を「柱に衝突しない」範囲まで縮めること。ただし柱1本だけ
チェックしても不十分(木の周りには柱が格子状に何本もある)なので、θ を
細かく走査して全周にわたる最小クリアランスを求める、という総当たりの
検証をそのままチューニング手順として採用した(`ORBIT_RADIUS_EUC=0.95`,
`ORBIT_RADIUS_HYP=0.38` で安全マージン確保)。カメラを前進させる罠4は
「視野から外れる」という見た目の不具合だったが、今回は「幾何的に無効な
状態(固体の内部)に入り込む」という別種の不具合で、レイマーチ系の
プロトタイプでは両方とも典型的に起こりうる、という教訓。

## 曲率のパラメータ化

右ペインの曲率を K として連続パラメータ化し、スライダーで K<0(双曲空間)
・K=0(ユークリッド空間)・K>0(球面)をシームレスに行き来できるようにした。
「双曲版」「ユークリッド版」を別々に実装するのではなく、両方(および球面)
を含む1本の数式群を作り、K の値だけで具体的な幾何が決まる、という設計。

### 一般化三角関数と一般化内積

鍵になるのは、球面幾何の `cos/sin` と双曲幾何の `cosh/sinh` を統一する
**一般化三角関数**:

```
cosK(x) = cos(√K·x)         (K>0)     cosK(x) = 1     (K=0)     cosK(x) = cosh(√-K·x)   (K<0)
sinK(x) = sin(√K·x)/√K      (K>0)     sinK(x) = x     (K=0)     sinK(x) = sinh(√-K·x)/√-K (K<0)
```

これらは常に恒等式 `cosK(x)² + K·sinK(x)² = 1` を満たす(K=1 で
`cos²+sin²=1`, K=-1 で `cosh²-sinh²=1` になる、おなじみの式の一般化)。
測地線に沿った移動は、これまで `P'=P·cosh(Δ)+d·sinh(Δ)` と書いていた式が
そのまま

```
P' = P·cosK(Δ) + d·sinK(Δ)
d' = P·(-K)·sinK(Δ) + d·cosK(Δ)
```

に一般化される(導出は「点 P, 方向 d に対する2階線形常微分方程式
`P''=-K·P, P'=d` の解」という、K=-1 のときの元の式と全く同じ立て方をする
だけでよい)。

内積(mdot)も一般化が要る。時間成分の重みに `1/K` を持たせた

```
gdot(a,b) = a.t·b.t / K + (空間成分の通常の内積)
```

を使うと、位置ベクトル P は常に `⟨P,P⟩=1/K`、方向ベクトル d は常に
`⟨d,d⟩=1` という規約が、K の符号によらず一貫して保てる(K=-1 で
`⟨P,P⟩=-1` になり、元の mdot と完全に一致する)。K>0 では 1/K の符号が
反転して自動的に「通常のユークリッド内積」として働き、|K|→0 では時間成分
の寄与がそのまま消えて「空間成分だけの通常の内積」に自然に一致する
——ユークリッド空間で時間軸(埋め込みの4次元目)が意味を持たないのと
ちょうど対応している。

### 床・柱・木の一般化

- **床**(全測地的超平面)までの距離は `asinK(gdot(p,n))` (n=法線)。
  `asinK` は1階微分が K=0 でも有限なので、桁落ちせず全ての K で安定に
  動く一本の式のまま繋がる。
- **柱**(測地線までの距離)と**木**(測地線分=カプセルまでの距離)は
  `cosK(距離) = √(ca² + K·cb²)` (ca, cb は p と測地線の基底ベクトルとの
  gdot)という式に一般化される。ただしこちらは K→0 で `cosK(d)→1` に
  潰れてしまい(1階微分が0のため)、距離を逆算する際に桁落ちする。この
  「点・線・カプセルまでの距離」だけは、`|K|` がごく小さい領域(実装では
  `K_EPS=0.02` 未満)で通常のユークリッド距離の式に明示的に切り替える
  分岐を入れている(床の asinK 系だけは分岐無しで繋がる、という非対称は
  この 1階微分の有無に由来する)。
- **柱の格子**は、以前の `x0=sinh(n·S), z0=cosh(n·S)·sinh(m·S)` が
  そのまま `x0=sinK(n·S), z0=cosK(n·S)·sinK(m·S)` に一般化される
  (導出は「原点にx方向の一般化ブースト→z方向の一般化ブーストを合成
  適用する」という元の導出と全く同じ)。
- **木**は K が変わるたびに JS 側で再生成し、フラットな `Float32Array`
  にして `uniform` 配列として毎フレーム渡す。以前は `const` 配列に
  焼き込んでいたが、それだと曲率を変えるたびにシェーダー再コンパイルが
  必要になり、スライダーがカクつく(このプロトタイプでは初回コンパイルに
  数十秒かかることもあるため、致命的に重い)。枝の本数は高々15本なので、
  JS 側の再計算コストは無視できる。

### K=0 での検算

`cosK`/`sinK`/`gdot`/`asinK` はどれも K=0 近傍で厳密にユークリッド版の
式へ一致するよう作ってある。スライダーを K=0 にすると、右ペインの見た目
(柱がまっすぐ、木の枝が直線的)が左ペイン(常に固定のユークリッド版)と
ほぼ同じ構図になることを実際に確認できる——これは「たまたま似ている」の
ではなく、式として厳密に同じものへ簡約されている。

### 曲率レンジを K∈[-1,1] に広げたときに踏んだ罠(木の除外ゾーン)

周回半径・柱の間隔・カメラ高さは、もともと K=-1(双曲空間)だけを基準に
チューニングした固定値だった。曲率レンジを K∈[-1,1] まで広げたところ、
K=-1 単体では問題無かった周回軌道が、その間の広い範囲(だいたい
`-0.5 < K < 1` 全域)で柱の内部にめり込んでしまうことが Node.js での
全周(θ)・全曲率(K)の総当たり走査で判明した。原因は2つ絡み合っている:

- **K→0 に近づくほど(負の側から)安全マージンが失われる**: 周回半径
  0.38 は、K=-1 の強い負の曲率が「実際の格子間隔以上の余裕」を生んで
  くれていたおかげで安全だった。曲率が弱まってほぼ平坦に近づくと、この
  余裕が消えて衝突が始まる。
- **K が正になると測地線が「折り返す」**: 球面上の測地線は、チャート
  座標で `n·SPACING·√K` が `π/2` に近づくと(大円の反対側へ向かい始め)
  符号が反転する。柱の格子はチャート座標を単純に伸ばしているだけなので、
  K が大きいところでは同じ添字の柱が想定と違う位置に「折り返って」現れ、
  一見無関係に見える角度で衝突を起こす。

半径や間隔を場当たり的に小さくしても、この2つの効きが逆方向(片方は
半径を大きく、もう片方は小さくしたい)なので固定値では解決しなかった。
最終的な対策は、**木の根元から一定距離より近い柱を格子から丸ごと除外
する**というもの。三角不等式から、木を中心とする半径 `ORBIT_RADIUS` の
円周上のどの点も、木から `TREE_EXCLUSION_RADIUS` 以上離れた柱までは
少なくとも `TREE_EXCLUSION_RADIUS − ORBIT_RADIUS` の距離を必ず保つ
——個々の θ・K の組み合わせを逐一チューニングするのではなく、幾何的な
不等式で衝突そのものを起こりえなくする設計にした。`TREE_EXCLUSION_RADIUS
=0.9` は K∈[-1,1] 全域・全周で Node.js 検証済み(最小クリアランス
0.2185、十分な安全マージン)。

## カメラ高さについて(発見: 平行線角)

左右のペインでカメラの床からの高さをあえて揃えていない
(`CAM_HEIGHT_EUC=1.0`, `CAM_HEIGHT_HYP=0.22`)。理由は罠4とは別の、
もう一つの実装中の発見。

双曲空間では、超平面から双曲距離 h だけ離れた点から出るレイのうち、
「ある臨界角より浅い下向き」のものは、どれだけ延長してもその超平面に
二度と届かない(漸近的にある高さへ収束しながら遠ざかっていく)。これは
双曲幾何で有名な**平行線角(angle of parallelism)** そのもので、h が
大きいほど臨界角は急峻になる(`h→∞` で「ほぼ真下」しか届かなくなる)。

最初 `CAM_HEIGHT` を左右とも 1.0 に揃えていたところ、双曲側は画面の
ほぼ全域(視野角内の下向き成分では足りない)が床にも柱にも当たらず、
彼方まで抜けて空だけが映った。ユークリッド側は同じ高さ・同じ見下ろし角で
普通に地面が見えるのに、双曲側は「屈まないと床が視界に入らない」——
これはバグではなく上記の平行線角が働いた結果で、`CAM_HEIGHT_HYP` を
0.2 前後まで下げてようやく実用的な構図になった。

## 格子の定義について(意図的な妥協)

格子は真に正則な双曲タイル張り(regular hyperbolic tessellation)では
ない(罠1のブースト合成方式も、あくまで「原点から見て行・列ともに歪みが
小さい」実用上の折衷案であって、閉じたタイル張りの基本領域・鏡映群を
使ったものではない)。利点は、ユークリッド版とまったく同じ添字グリッド
`(i,j)` を両方の幾何で解釈でき、「同じ格子を2つの曲率で見る」という
今回の目的に都合が良いこと。また通常の `floor` によるドメイン
リピティション(現在の marched point 周辺の ±K セルだけを調べる)が
そのまま使える。

## つまずいた点・チューニングのメモ(その他)

- **float32 精度と `acosh` のキャンセレーション**: 前進距離 t とレイの
  到達距離を大きく取りすぎると、双曲面上の座標が `cosh/sinh` で指数的に
  増大し、`cylSDF` 内の `pa*pa - pb*pb`(ほぼ等しい大きな数同士の引き算)
  で桁落ちが起き、ノイズだらけの絵になった。`HYP_MAX_DIST=6.0` 程度に
  抑え、`FOG_DENSITY_HYP` を強めにして遠方を早めに霧へ溶かすことで
  実用範囲に収めている(罠4の対策で `T_MAX` 自体も縮めたため、この問題は
  今はほぼ顕在化しない)。
- **デバッグの実践**: GLSL の中身を直接デバッグするのは難しいので、
  疑わしい数式を Node.js へそのまま移植し、倍精度で追跡する、という
  手順が終始有効だった。特に「hit フラグは正しいのに絵がおかしい」
  ように見えたときは、`outColor` に `hit`・法線・面の種類(床/柱)などを
  直接生の値として出力し、シェーディングを迂回して二分探索的に原因箇所を
  絞り込んだ。
- **木の再帰段数とシェーダーのコンパイル時間**: フラクタルな木の再帰段数
  (`TREE_CONFIG.MAX_DEPTH`)を素朴に 4(枝31本)にしたところ、ヘッドレス
  Chromium + SwiftShader(ソフトウェアレンダラ)上でのシェーダーコンパイル
  が数十秒〜1分以上かかるようになった。`map()` がループ内で
  `pillarSDF`(9候補)と `treeSDF`(N本)を毎回評価し、それが `MAX_STEPS`
  回のスフィアトレースと `calcNormal` の中心差分(`map` をさらに6回)から
  呼ばれるため、コンパイラが積極的にインライン展開すると生成コードが
  一気に膨らむ。実機の GPU ドライバはずっと賢いはずだが、安全側に振って
  `MAX_DEPTH=3`(枝15本)まで下げた。見た目のフラクタル感は保ったまま、
  体感できるレベルまでコンパイル時間が縮んだ。

## 分かったこと・検証方法について

Playwright(Chromiumヘッドレス、`--use-gl=angle --use-angle=swiftshader`)
でシェーダーのコンパイルエラーとコンソールエラーの有無を確認しつつ、
実際にスクリーンショットを撮って視覚的に検証した。この実装で踏んだ罠は
すべて、コンソールにはエラーが出ない(GLSL 的には正しくコンパイル・
リンクされ、実行時エラーも起きない)類のもので、スクリーンショットでの
目視確認と、疑わしい計算を Node.js に移植して倍精度で追跡する手順の
組み合わせが無ければ、どれも気づけなかった。

## 次のステップ

- 真に正則な双曲タイル張り(例えば {4,5} タイリング)による格子への
  差し替え。現状のブースト合成グリッドは「同じ添字を2つの曲率で見る」
  という比較目的には適しているが、双曲空間本来の意味での正則なタイル
  張りとは異なる。
- 周回半径(`ORBIT_RADIUS_HYP`)を大きく取ると、罠4と同種の「対象までの
  距離自体は一定でも、格子の遠い部分は見え方が安定しない」現象が別の形で
  再燃する可能性がある。現状の半径では未検証。
- 曲率スライダーの範囲をさらに拡張する場合(現状 K∈[-1,1])、木の除外
  ゾーンだけでは対応しきれなくなる可能性がある。特に K が `1` を大きく
  超えると、柱の格子そのもの(木から離れた場所)が測地線の折り返しで
  自己交差を始める(「曲率のパラメータ化」節参照)。より広い範囲まで
  安定させるには、K に応じて格子間隔そのものを動的に縮小する対応が要る。
- 遠方まで安定して探索できるよう、座標の周期的な再正規化または
  倍精度シミュレーションの導入。
- モバイル実機での操作感(現状はドラッグでの見回し + 自動周回のみ、
  タッチでの周回速度操作は未対応)の確認。曲率スライダーのタッチでの
  操作感も併せて確認したい。
